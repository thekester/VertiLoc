0. RÉSUMÉ EXÉCUTIF
════════════════════════════════════════════════════════════════════════════════

  Ce rapport documente l'ensemble des benchmarks réalisés pour un système de
  localisation indoor par fingerprinting RSSI Wi-Fi à granularité cellule
  (25 cm × 30 cm), sans capteur supplémentaire, dans 4 salles distinctes.

  CHIFFRES CLÉS POUR L'ARTICLE
  ──────────────────────────────

  Données :
  - 4 salles (B121, D005, E101, E102) | 29 730 échantillons total
  - 80/34/72/75 cellules | 5 features RSSI (Signal, Noise, A1, A2, A3)

  Meilleure précision (protocole room-aware, multi-salles, n=29 730) :
  - Top-1 (cellule exacte)    : 91.8 %  (KNN raw)      |  erreur moy. 0.097 m
  - Top-3 spatial             : 96.9 %  (NN+L-KNN)     |  top-3 min error 0.025 m
  - p90 erreur                : 0.00 m  → 90 %+ des prédictions sont à 0 cm
  - Précision distance (embedding NN+L-KNN) : 94.7 %

  Précision intra-salle (sans one-hot salle) :
  - E102 (75 cellules, 6 campagnes) : KNN = NN+L-KNN = 92.7 %
  - E101 (72 cellules, 6 campagnes) : KNN 91.0 %

  Généralisation — limites fondamentales :
  - LORO (salle inconnue)         : ~1–2 % tous modèles  → non transférable
  - LOCO généralisé (B121/E101/E102) : HistGB 1.82–2.42 % | aléatoire ~1.25–1.39 %
  - LOCO E102 (campagne inconnue) : HistGB 2.42 %         → non transférable
  - LOCO B121 (campagne inconnue) : HistGB 2.30 %         → non transférable
  - B121/deplacement (inférence OOD/mouvement) : 100 % rejet OOD ; 0 % exact sur 6 points statiques
  - Conclusion : chaque déploiement requiert une recalibration complète.

  Robustesse :
  - Bruit synthétique σ=1 dBm : perte < 1 pp (KNN, LOCO E102)
  - Bruit σ=3 dBm             : dégradation significative (voir section 16septies)
  - Learning curve : 60 % des données suffisent pour atteindre ~90 % du plateau

  Protocole TriScope (modèle multi-tâche, D005+E101, 50 tirages) :
  - Cellule : 97.6 % ± 3.4 % | Distance routeur : 57.1 % | Salle : 72.1 %

  Protocole 360° XGBoost/RF (E101 circulaire + E102 orientations, 90/10) :
  - Meilleur modèle testé : XGBoost cellule directe, accuracy 72.8 %
  - Erreur médiane : 0.00 m | p80 : 0.590 m | p90 : 1.494 m
  - La régression polaire (distance + sin/cos) est moins performante que la
    classification directe de cellule sur ces données.

  Calibration / Incertitude :
  - ECE KNN (brut) : 0.023 → très bien calibré naturellement
  - HistGB + temperature scaling : ECE 0.012, NLL 0.285 (meilleure calibration)

  Modèle recommandé en production :
  - Décisions "hard" (argmax)     → KNN (k=7, poids distance)
  - Décisions probabilistes       → HistGB + temperature scaling
  - Contrainte mémoire/latence    → voir section profil embarqué (16sexies/16quinquiesdecies)

  Référence état de l'art (Wi-Fi fingerprinting) :
  - Travaux publiés (KNN RSSI indoor) : 70–90 % @ grille 0.5–1 m typique
  - Ce système : 92–94 % @ cellule 25×30 cm → granularité ×20 plus fine

  Approches avancées testées (avril 2026 — résultats section 18) :
  - TabNet (pytorch-tabnet) : 25 % room-aware → inadapté au problème 5D
  - CORAL + DANN (adaptation domaine) : LORO inchangé (~1.2–1.3 %) → résultat négatif solide
  - Metric Learning : équivalent à KNN en room-aware ; léger gain E102 (92.8 %)
  - SAE + KNN latent : conserve KNN ; léger gain E102 (92.76 %, 10.8 cm)
  - SSGAN + TabDDPM : pas de gain LOCO E102, augmentation synthétique non retenue
  - TabM : 38.1 % room-aware / 33.1 % E102 → sous KNN/RF
  - Few-shot cible LORO : TargetKNN atteint 48.9 % en K=5, 66.9 % en K=10,
    84.7 % en K=25 ; ProtoNet plafonne à ~17 %
  - Few-shot cible LOCO : TargetKNN atteint 89.7 % en K=5, 94.6 % en K=10,
    96.2 % en K=25 en moyenne sur B121/D005/E101/E102
  - TTA / Tent : ne corrige pas LORO (1.43 % vs 1.71 % vanilla)
  - DeepSets : échec sur split anti-leakage strict (≤3.3 %)
  - GNN cellules : échec pratique (2.6–2.8 %)
  - Self-supervised MAE/DAE/SimCLR : utile seulement en embedding + KNN
  - Spatially-aware loss : baisse l'erreur MLP mais baisse l'accuracy
  - Mixture-of-Experts : améliore MLP (MoE-4 : 46.5–50.4 %) sans atteindre KNN
  - FOMAML : reste au plancher LORO (~1.6–1.9 %)
  - Conformal prediction : couverture ~90 % à α=10 %, ensembles 17–20 cellules
  - TCN-PN (Ma & Shi, Sensors 2023) — 2 variantes testées (26 avr. 2026) :
      · TCN fidèle au papier (v1) : K=1 → 17.3 %/1.056 m ≈ ProtoNet-MLP ;
        accuracy DÉCROÎT avec K → architecture TCN inadaptée aux 5 features
      · MLP+confiance (v2, contribution clé adaptée) : acc CROÎT avec K ;
        K=25 → 28.9 %/0.926 m = meilleur ProtoNet few-shot LORO testé ;
        gain vs ProtoNet vanilla : +4.6 pp, −53 mm à K=25
      → MLP_conf recommandé ; TCN causal dilué inutile sur vecteur 5D

1. CONTEXTE ET OBJECTIF
════════════════════════════════════════════════════════════════════════════════

L'objectif du projet est de réaliser une localisation indoor fine-grained (à
l'échelle de la cellule de 25 cm × 30 cm) en utilisant uniquement les signaux
Wi-Fi (RSSI) mesurés depuis un point d'accès fixe (routeur). Aucun capteur
supplémentaire n'est utilisé.

L'approche retenue est le fingerprinting RSSI : une phase hors-ligne ("offline")
collecte des mesures RSSI pour chaque position connue de la grille, constituant
une "carte radio". En phase en-ligne ("online"), le système classifie une
nouvelle mesure RSSI pour prédire la cellule occupée.

Matériel utilisé :
  - Routeur Wi-Fi standard avec 3 antennes distinctes (A1, A2, A3)
  - PC de mesure relié au routeur par câble RJ45 (distance routeur-PC : 125 cm)
  - Distance routeur–tableau blanc : 4 mètres, centré sur le tableau

Grille de mesure :
  - Cellules au format X_Y (ligne_colonne), taille 25 cm (largeur) × 30 cm (hauteur)
  - Lignes 0 à 4 (de devant vers le fond de la salle)
  - Colonnes de -4 à +10 (gauche à droite)
  - 75 cellules en E102, 35 cellules en D005, 34-72 cellules en E101


════════════════════════════════════════════════════════════════════════════════
2. FEATURES (VARIABLES D'ENTRÉE)
════════════════════════════════════════════════════════════════════════════════

Chaque échantillon est composé de 5 features brutes issues du scan Wi-Fi :

  1. Signal   : RSSI global du point d'accès (dBm)
  2. Noise    : niveau de bruit ambiant (dBm)
  3. signal_A1 : RSSI de l'antenne 1 du routeur (dBm)
  4. signal_A2 : RSSI de l'antenne 2 du routeur (dBm)
  5. signal_A3 : RSSI de l'antenne 3 du routeur (dBm)

Ces 5 features constituent le vecteur d'entrée des modèles de classification.
Dans les configurations "room-aware", un vecteur one-hot encodant la salle est
concaténé aux features RSSI. Dans le modèle NN+L-KNN, une couche d'embedding
apprend une représentation latente avant la classification KNN.

Nombre d'échantillons par cellule : 25 mesures par campagne.

4. MODÈLES DE MACHINE LEARNING ÉVALUÉS
════════════════════════════════════════════════════════════════════════════════

Tous les modèles résolvent un problème de classification multi-classes (75 classes
= 75 cellules) à partir du vecteur RSSI de 5 dimensions.

────────────────────────────────────────────────────────────────────────────────
4.1 K-Nearest Neighbors (KNN)
────────────────────────────────────────────────────────────────────────────────
  Principe : classe prédite = majorité parmi les k voisins les plus proches
  (distance euclidienne dans l'espace RSSI). Paramètre k optimisé par CV.
  Variantes testées :
    - KNN raw : KNN direct sur les 5 features
    - KNN distance+scaled : KNN pondéré par la distance après standardisation
      des features (baseline simple et robuste)
    - Bagging KNN : ensemble de KNN sur sous-échantillons bootstrap
    - KNN Mahalanobis : distance de Mahalanobis (skippé, données insuffisantes)
    - LDA+KNN : projection LDA puis KNN dans l'espace réduit

────────────────────────────────────────────────────────────────────────────────
4.2 NN + L-KNN (Neural Network + Latent KNN)
────────────────────────────────────────────────────────────────────────────────
  Architecture hybride en deux étapes :
    1. Un réseau de neurones (MLP) apprend un embedding latent des features RSSI
    2. Un KNN est appliqué dans cet espace latent pour la classification finale
  Avantage : capture les non-linéarités du signal RSSI tout en conservant la
  robustesse du KNN. Dans la configuration "cell+distance", le NN prédit
  simultanément la cellule ET la distance routeur.

────────────────────────────────────────────────────────────────────────────────
4.3 Random Forest (RF)
────────────────────────────────────────────────────────────────────────────────
  Ensemble de N arbres de décision entraînés sur des sous-échantillons bootstrap
  et des sous-ensembles de features. Robuste au sur-apprentissage.

────────────────────────────────────────────────────────────────────────────────
4.4 Extra Trees (ET)
────────────────────────────────────────────────────────────────────────────────
  Variante de Random Forest avec seuils de décision aléatoires (Extremely
  Randomized Trees). Généralement plus rapide que RF, précision comparable.

────────────────────────────────────────────────────────────────────────────────
4.5 Gradient Boosting (GBDT)
────────────────────────────────────────────────────────────────────────────────
  Ensemble séquentiel d'arbres corrigeant les erreurs du modèle précédent.
  Variantes :
    - GBDT classique (sklearn GradientBoostingClassifier)
    - HistGradientBoosting (HistGB) : version optimisée par histogrammes,
      inspirée de LightGBM, très rapide sur grands datasets

────────────────────────────────────────────────────────────────────────────────
4.6 Support Vector Machine (SVM)
────────────────────────────────────────────────────────────────────────────────
  Classifieur maximisant la marge entre classes. Noyaux testés :
    - SVM RBF (noyau gaussien)
    - SVM linéaire
    - SVM polynomial (degré 3)

────────────────────────────────────────────────────────────────────────────────
4.7 Deep MLP Softmax
────────────────────────────────────────────────────────────────────────────────
  Réseau de neurones profond (sklearn MLPClassifier) avec activation softmax
  en sortie pour la classification multi-classes.

────────────────────────────────────────────────────────────────────────────────
4.8 Modèles probabilistes
────────────────────────────────────────────────────────────────────────────────
  - GaussianNB : Naive Bayes gaussien (hypothèse d'indépendance des features)
  - QDA : Analyse Discriminante Quadratique
  - LDA : Analyse Discriminante Linéaire (utilisée comme prétraitement pour KNN)
  - Logistic Regression calibrée (Platt/sigmoid via CV) :
    baseline discriminative multi-classes avec probabilités calibrées.

────────────────────────────────────────────────────────────────────────────────
4.9 Modèle Multihead (RSSI seul, sans label de salle)
────────────────────────────────────────────────────────────────────────────────
  Architecture multi-tâche prédisant simultanément :
    - La cellule (classification à 75 classes)
    - La distance routeur (classification à 4 classes : 2/3/4/5 m)
    - La salle (classification à 3 classes : D005/E101/E102)
  Depuis le vecteur RSSI brut uniquement, sans one-hot salle.

────────────────────────────────────────────────────────────────────────────────
4.10 Modèles optionnels / dépendances variables
────────────────────────────────────────────────────────────────────────────────
  - CatBoost, LightGBM : skippés dans les benchmarks principaux lorsque les
    dépendances ne sont pas installées dans l'environnement courant.
  - XGBoost : non inclus dans le benchmark principal multi-salles historique,
    mais utilisé dans le benchmark auxiliaire 360° exécuté depuis la `.venv`
    locale (voir section 16novemdecies).
  - Stacking (RF+SVM+MLP) : skippé dans les runs finaux (temps de calcul excessif,
    ~66 min par fold sur le dataset multi-salles)

────────────────────────────────────────────────────────────────────────────────
4.11 Piste hybride physique + ML (orientation Sionna RT)
────────────────────────────────────────────────────────────────────────────────
  Une piste complémentaire a été ajoutée pour relier modèle physique et
  adaptation data-driven, dans l'esprit Sionna RT (ray tracing différentiable).

5. PROTOCOLES D'ÉVALUATION
════════════════════════════════════════════════════════════════════════════════

────────────────────────────────────────────────────────────────────────────────
5.1 Room-aware (intra-salle / multi-salle calibré)
────────────────────────────────────────────────────────────────────────────────
  Principe : toutes les données des salles disponibles sont mélangées, puis
  splitées 80% train / 20% test de façon stratifiée. Le modèle reçoit en entrée
  les features RSSI + un vecteur one-hot identifiant la salle.
  Interprétation : performance attendue en conditions réelles APRÈS calibration
  complète de toutes les salles. Le modèle connaît dans quelle salle il opère.
  Ce protocole ne teste PAS la généralisation à une nouvelle salle.

────────────────────────────────────────────────────────────────────────────────
5.2 LORO — Leave-One-Room-Out (généralisation inter-salles)
────────────────────────────────────────────────────────────────────────────────
  Principe : pour chaque salle R, entraîner sur toutes les autres salles,
  tester sur R. Avec l'ajout de B121, le protocole couvre désormais 4 folds
  (B121, D005, E101, E102 tour à tour).
  Interprétation : capacité du modèle à localiser dans une NOUVELLE salle non
  calibrée. C'est le vrai test de généralisation du fingerprinting RSSI.
  Limite fondamentale : les RSSI d'une salle inconnue sont incompatibles avec
  la carte radio d'une autre salle (propagation, géométrie, obstacles différents).

────────────────────────────────────────────────────────────────────────────────
5.3 LOCO — Leave-One-Campaign-Out (robustesse intra-salle)
────────────────────────────────────────────────────────────────────────────────
  Principe : dans une salle donnée, retirer une campagne entière, entraîner sur
  les autres et tester sur la campagne retirée.
  Interprétation : robustesse du modèle à un changement de condition opératoire
  au sein d'une même salle calibrée partiellement.
  Cas couverts dans ce rapport :
  - E102 : 6 campagnes (orientation antenne + élévation)
  - E101 : 4 orientations circulaires (back/front/left/right)
  - B121 : 3 campagnes (routeur centre/gauche/droite à 3 m)

────────────────────────────────────────────────════════════════════════════════
5.4 Classification de distance routeur
────────────────────────────────────────────────────────────────────────────────
  Tâche séparée : prédire la distance routeur–tableau (2/3/4/5 m) depuis le
  vecteur RSSI brut. Testé sur le dataset multi-salles (B121+D005+E101+E102).

────────────────────────────────────────────────────────────────────────────────
5.5 Classification de salle (room identification)
────────────────────────────────────────────────────────────────────────────────
  Tâche séparée : prédire la salle (B121/D005/E101/E102) depuis l'embedding appris
  par NN+L-KNN (sans label de salle). LogisticRegression sur l'embedding.

────────────────────────────────────────────────────────────────────────────────
5.6 Analyse spatiale (E102, RF 5-fold CV)
────────────────────────────────────────────────────────────────────────────────
  Évaluation cellule par cellule via cross-validation 5 folds sur E102 (RF).
  Permet d'identifier les zones géographiques difficiles à localiser.

────────────────────────────────────────────────────────────────────────────────
5.7 Protocole anti-leakage strict (à implémenter/valider)
────────────────────────────────────────────────────────────────────────────────
  Objectif : garantir qu'aucun quasi-duplicat (même cellule/campagne, RSSI très
  proche) n'est partagé entre train et test.

  Règles proposées :
  1) Split par groupes atomiques :
     - groupe = (room, campaign, grid_cell)
     - un groupe appartient intégralement au train OU au test, jamais aux deux.
  2) Vérification post-split obligatoire :
     - intersection exacte des groupes train/test = vide ;
     - intersection des signatures RSSI quantifiées (ex. pas 0.5 dB) sur un même
       (room, campaign, grid_cell) = vide.
  3) Agrégations calculées train-only :
     - toute feature agrégée (moyenne/variance/prototype par cellule/campagne,
       normalisation globale, etc.) est fit uniquement sur train puis appliquée
       au test ;
     - interdiction de recalculer une statistique en mélangeant train+test.
  4) Audit de fuite traçable :
     - journaliser `n_groups_train`, `n_groups_test`, `n_overlap_groups`,
       `n_overlap_signatures` ;
     - si overlap > 0, run invalide (échec explicite du benchmark).

  Bénéfice : métriques plus conservatrices mais robustes, et comparaison
  inter-modèles plus fiable en présence de séries RSSI fortement redondantes.

────────────────────────────────────────────────────────────────────────────────
5.8 Top-k spatial (ajout opérationnel)
────────────────────────────────────────────────────────────────────────────────
  En complément des métriques classiques (accuracy top-1, erreur moyenne),
  on suit désormais :
  - top-1 accuracy (`top1_acc`) ;
  - top-3 accuracy (`top3_acc`) ;
  - distance au meilleur candidat parmi le top-3 (`top3_best_error_m`) ;
  - gain spatial top-3 vs top-1 (`top3_best_gain_m`), en mètres.

  Intérêt opérationnel :
  même quand la cellule exacte top-1 est ratée, le top-3 peut rester
  géographiquement proche de la vérité terrain, ce qui reflète mieux
  la "rattrapabilité" de la décision en exploitation.

11quater. B121/DEPLACEMENT — CHALLENGE D'INFÉRENCE HORS ENTRAÎNEMENT
════════════════════════════════════════════════════════════════════════════════

Objectif :
  Évaluer directement le modèle d'inférence exporté sur un dossier explicitement
  exclu de l'entraînement (`data/B121/deplacement`) afin de tester sa capacité à
  "s'en sortir" sur des positions hors grille, des trajectoires en mouvement et
  des placements sémantiques (fond/centre/gauche/droite de la pièce).

Contenu évalué :
  - 6 points statiques : `0_15`, `2_-3`, `2_2`, `3_-8`, `3_9`, `4_12`
  - 10 traces d'inférence seules : `rond_411`, `diag_44`, `diag_4-4`,
    `hauteur_47`, `b_c`, `b_d`, `b_g`, `c_c`, `c_d`, `c_g`
  - 25 mesures RSSI par fichier, soit 16 traces × 25 = 400 inférences

Protocole :
  - Aucun ré-entraînement ; utilisation directe de `reports/localizer.joblib`
  - Inférence batch via `VertiLocInferenceModel`
  - Lecture des scores OOD déjà embarqués (OOD = out-of-distribution,
    c'est-à-dire détection d'échantillons dont la signature RSSI s'écarte de la
    distribution radio observée pendant l'entraînement)
  - Recalibration du seuil OOD en distance latente :
    le seuil n'est plus estimé à partir du seul 2e voisin d'entraînement,
    mais du plus proche voisin associé à une cellule différente, afin
    d'éviter une sous-estimation due à des répétitions quasi identiques
    au sein d'une même cellule

Résultats principaux :
  - Le taux moyen de rejet OOD tombe à 4.25 %
    (17/400 échantillons), avec rejet non nul sur 2 traces seulement :
    `4_12` = 36 %, `b_c` = 32 %
  - Points statiques : 0 % de prédiction exacte sur les 6 cas
  - Erreur dominante sur les 6 points statiques : moyenne 1.884 m,
    médiane 1.583 m, max 3.856 m
  - Même le point interne `2_2` n'est pas récupéré exactement :
    cellule dominante prédite = `1_2` (erreur 0.30 m)
  - Les points hors tableau (`0_15`, `3_-8`, `4_12`) sont rabattus vers des
    cellules internes, jamais correctement identifiés comme position géométrique
    extrapolée
  - Les trajectoires (`rond`, `diag`, `hauteur`) et placements sémantiques
    (`b_*`, `c_*`) restent projetées vers des cellules d'entraînement connues plutôt que
    reconstruites comme mouvement continu ou position extrapolée

Interprétation :
  Ici, le score OOD ne mesure pas l'exactitude géométrique de la prédiction,
  mais le degré d'écart entre un échantillon RSSI et le support radio appris.
  Un échantillon marqué OOD est donc interprété comme "hors de la distribution
  d'entraînement", pas nécessairement comme "faux" au sens strict, tandis qu'un
  échantillon accepté n'est pas nécessairement bien localisé. La recalibration
  du seuil en distance latente porte ce seuil à `30.514` et rétablit un
  comportement d'inférence cohérent sur les traces évaluées.

  La conclusion sur la généralisation demeure toutefois inchangée. Le système
  se comporte comme un classifieur fermé sur sa carte radio d'entraînement :
  les RSSI hors grille ou en mouvement sont rabattus vers les cellules connues
  les plus proches, sans extrapolation géométrique fiable ni suivi continu
  robuste. Le point `2_2` reste proche de sa zone cible (`1_2`, erreur 0.30 m),
  mais aucun des 6 points statiques n'est retrouvé exactement, et les cas les
  plus éloignés (`0_15`, `3_-8`, `4_12`) restent mal localisés.

  En pratique, cette expérience montre que la limite principale du modèle
  d'inférence est son incapacité à sortir du cadre de son entraînement. Face à
  des positions hors grille, des trajectoires continues ou des configurations
  sémantiques non vues, il ne généralise pas réellement : il rabat les mesures
  vers les cellules apprises les plus proches au lieu de produire une
  extrapolation spatiale fiable.

Faits saillants (sur 6 folds LOCO) :
  - Le split le plus décalé est exp6_1m50 (routeur surélevé), avec les plus
    fortes valeurs de divergence :
      MMD = 0.1827, Wasserstein_mean = 3.335, PSI_mean = 0.958, PSI_max = 2.003.

    élévation forte (exp6) > changements d'orientation simples.

Corrélation shift -> dégradation de performance :
  - Cible 1 : `acc_drop` = max(acc) - acc (par modèle, par fold),
  - Cible 2 : `err_increase` = err - min(err) (par modèle, par fold).

Exemples de corrélations observées (Pearson) :
  - ExtraTrees, `err_increase` vs `psi_max` : r = +0.820

Interprétation :
  - Sur plusieurs modèles (notamment ExtraTrees/HistGB), plus le shift
    train/test est élevé, plus l'erreur augmente.
  - Le signal est net pour l'erreur moyenne, plus instable pour l'accuracy
    cellule (normal sur un problème à 75 classes déjà proche du hasard en LOCO).
  - Taille d'échantillon faible (n=6 folds) : ces corrélations sont des indices
    mécanistiques utiles, pas une preuve statistique définitive.

16. SYNTHÈSE ET CONCLUSIONS
════════════════════════════════════════════════════════════════════════════════

────────────────────────────────────────────────────────────────────────────────
14.1 Performance intra-salle (conditions calibrées)
────────────────────────────────────────────────────────────────────────────────
  Les meilleures approches (Bagging KNN, LDA+KNN, NN+L-KNN, HistGB) atteignent
  93–94 % de précision cellule avec erreur <9 cm sur les multi-salles calibrées,
  et 92–93 % sur E102 seul. Le p90=0 confirme que la majorité des prédictions
  sont exactes au cm près.

────────────────────────────────────────────────────────────────────────────────
14.2 Limite fondamentale : non-transférabilité inter-salles
────────────────────────────────────────────────────────────────────────────────
  Le LORO démontre que TOUS les modèles échouent (~1–2 %) lorsque la salle
  cible n'a pas été calibrée. L'empreinte RSSI est propre à chaque environnement
  physique. Cette limitation est inhérente au fingerprinting et non aux modèles.

────────────────────────────────────────────────────────────────────────────────
14.3 Sensibilité aux conditions opératoires (LOCO)
────────────────────────────────────────────────────────────────────────────────
  Au sein d'E102, un changement d'orientation d'antenne ou d'élévation du
  routeur est aussi dévastateur qu'un changement de salle (~1–2 %). La campagne
  au sol (exp5) est particulièrement incompatible avec les autres conditions.
  → Pour un déploiement robuste, il est nécessaire de couvrir toutes les
    configurations opératoires dans les données de calibration.

────────────────────────────────────────────────────────────────────────────────
14.4 Hiérarchie des modèles
────────────────────────────────────────────────────────────────────────────────
  TIER 1 (≥93 %) : Bagging KNN, LDA+KNN, NN+L-KNN, HistGB, KNN raw
  TIER 2 (90–93 %): ExtraTrees, RandomForest
  TIER 3 (83–90 %): GradientBoosting
  HORS-CLASSE (<50 %): SVM (RBF/poly/lin), Deep MLP sklearn, GaussianNB, QDA

  Les modèles à noyau (SVM) et les modèles purement paramétriques (MLP sklearn,
  GaussianNB) sont inadaptés à ce problème de classification RSSI à 75 classes.

────────────────────────────────────────────────────────────────────────────────
14.5 Recommandations pour les prochaines campagnes
────────────────────────────────────────────────────────────────────────────────
  1. Inclure systématiquement TOUTES les orientations d'antenne prévues en
     déploiement dans le dataset de calibration (le LOCO le confirme).
  2. L'élévation (au sol vs. standard vs. surélevé) requiert des campagnes
     dédiées si le routeur peut être placé à différentes hauteurs.
  3. La cellule 1_-4 (coin gauche, ligne 1) est systématiquement difficile :
     envisager une densification de la grille dans les coins.
  4. Modèles recommandés : KNN-based (Bagging KNN ou LDA+KNN) pour leur
     simplicité et leur performance maximale. NN+L-KNN pour les scénarios
     où la distance doit aussi être estimée (91.9 % de précision distance).

────────────────────────────────────────────────────────────────────────────────
14.6 Piste complémentaire : GAN / CycleGAN / estimation générative de carte radio
────────────────────────────────────────────────────────────────────────────────
  Les approches génératives récentes (GAN conditionnels, CycleGAN, generative
  radio map estimation) constituent une piste crédible pour réduire le coût de
  recalibration lorsqu'une nouvelle salle dispose de peu de mesures.

  Intérêt principal :
  - Générer/compléter des fingerprints RSSI manquants à partir d'un sous-ensemble
    de points mesurés.
  - Effectuer un transfert de domaine (salle source -> salle cible) en apprenant
    une transformation de distribution RSSI.

  Limites pratiques (premier déploiement) :
  - Entraînement instable et sensible au tuning (mode collapse, variabilité run-to-run).
  - Risque d'halluciner des structures radio non physiques si la contrainte
    géométrique/proagation n'est pas suffisamment imposée.
  - Complexité d'implémentation/validation nettement supérieure à RadioUNet,
    XGBoost ou un GNN bien régularisé.

  Position recommandée pour ce projet :
  - Prioriser d'abord un socle robuste et reproductible (RadioUNet, XGBoost,
    GNN) comme baseline opérationnelle.
  - Évaluer ensuite la piste générative comme module d'augmentation/transfert,
    avec protocole strict :
      (i) scénario low-data cible (5-20 % de points mesurés),
      (ii) comparaison contre interpolation + baselines discriminatives,
      (iii) métriques cellule + erreur spatiale + stabilité inter-seeds,
      (iv) ablation avec/sans contraintes physiques.

  Conclusion opérationnelle : piste prometteuse pour le few-shot inter-domaines,
  mais plus risquée qu'une approche discriminative classique en premier essai.

────────────────────────────────────────────────────────────────────────────────
14.7 Essais implémentés (code) — cGAN/CycleGAN sur E102 (low-data)
────────────────────────────────────────────────────────────────────────────────
  Implémentation réalisée : script `scripts/generative_radio_map.py` avec deux
  modes (`cgan`, `cyclegan`) et protocole d'évaluation standardisé :
  - baseline : KNN entraîné sur faible quantité de données cible
  - augmenté : KNN entraîné sur (données réelles low-data + données synthétiques)
  - upper bound : KNN entraîné sur train cible complet

  Paramétrage principal des essais :
  - cGAN : 40 epochs, 20 échantillons synthétiques/cellule
  - CycleGAN : 20 epochs, transfert exp2 -> cible, 20 synthétiques/cellule
  - split cible : 80/20, low-data = 10 % ou 20 % du train

  Résumé chiffré (delta = augmenté - baseline) :
  - cGAN (6 runs) :
    delta_acc moyen  = -0.09 points
    delta_err moyen  = +0.005 m
    Observation : effet global quasi neutre, parfois légère amélioration locale
    de l'erreur (ex: exp1 à 10 %, seed 7 : -0.005 m), mais non stable selon seed.
  - CycleGAN (3 runs) :
    delta_acc moyen  = -4.53 points
    delta_err moyen  = +0.066 m
    Observation : dégradation systématique de la précision cellule ; transfert
    de style RSSI inter-campagne insuffisamment fiable dans cette configuration.

  Stabilité statistique recommandée (idée à explorer en protocole standard) :
  - Répéter chaque protocole avec au moins 10 seeds indépendantes.
  - Reporter, pour chaque métrique (accuracy, erreur moyenne, delta), le triplet
    moyenne ± écart-type, puis l'IC95% :
      IC95% = moyenne ± t_(0.975, n-1) * (ecart_type / sqrt(n))
  - Interprétation : un gain n'est considéré robuste que si l'IC95% du delta
    n'inclut pas 0.
  - Sortie dédiée ajoutée dans le code : `generative_trials_stats.csv`
    (agrégation par protocole avec n_seeds, moyenne, std, bornes IC95%).
  - Exécution recommandée (10 seeds) :
    `python3 scripts/super_benchmarks.py --run-generative --generative-profile full --generative-seeds 1-10`

  Conclusion empirique à ce stade :
  - La piste générative est bien implémentée et testée, mais ne dépasse pas les
  baselines discriminatives sur les essais actuels.
  - cGAN peut être conservé comme module exploratoire (effet faible/neutre),
    tandis que CycleGAN est à considérer comme trop risqué sans contraintes
    physiques additionnelles ou tuning plus poussé.

  - Sur ce protocole homogène, KNN et NN+L-KNN dominent nettement le compromis

  Objectif :

  - Tests utilisés :

    * Paired t-test (erreur distance en m)
  - Dataset : complet, toutes salles + toutes distances (23 750 échantillons)
  - Split : 80/20 stratifié par cellule (n_test = 4 750 échantillons)
  - Sorties : `significance_room_aware.json/csv`

    Rang | Modèle      | Acc. cellule | Err. moy. (m) | p90 (m)
    ─────────────────────────────────────────────────────────────
      1  | KNN         |   92.04 %    |    0.105 m    |  0.000 m
      2  | NN+L-KNN    |   91.62 %    |    0.112 m    |  0.000 m
      3  | ExtraTrees  |   90.40 %    |    0.120 m    |  0.000 m

  Tests pairés (n_test = 4 750 échantillons) :

    Paire              | Δacc (pp) | McNemar p | Wilcoxon p | t-test p | Significatif ?
    ──────────────────────────────────────────────────────────────────────────────────
    KNN vs NN+L-KNN    |  +0.42 pp |   0.070   |   0.039 *  |  0.032 * | Marginal (err)
    KNN vs ExtraTrees  |  +1.64 pp |  <0.0001  |   0.004 ** |  0.003** | Oui (***)
    NN+L-KNN vs ET     |  +1.22 pp |   0.0002  |   0.135    |  0.123   | Oui (class.) / Non (err)

  (* p<0.05 ; ** p<0.01 ; *** p<0.001)

  Lecture :
  - KNN vs NN+L-KNN : la différence d'accuracy (+0.42 pp) n'est PAS
    statistiquement significative selon McNemar (p=0.070), mais l'erreur
    distance est légèrement meilleure pour KNN (Wilcoxon p=0.039).
    → Ces deux modèles sont pratiquement équivalents.
  - KNN vs ExtraTrees : différence fortement significative selon tous les

  - KNN et NN+L-KNN sont statistiquement indiscernables sur ce dataset.
    Le choix entre ces deux modèles doit se faire sur des critères
    opérationnels (latence, mémoire, interprétabilité).
  - La supériorité de KNN et NN+L-KNN sur ExtraTrees est statistiquement

16quarterdecies. IMPORTANCE DES FEATURES ET ABLATION ROOM-AWARE (E102, CV 5-fold)
════════════════════════════════════════════════════════════════════════════════

  Objectif :
  - Quantifier la contribution de chaque feature RSSI sur la précision cellule
    dans un contexte room-aware (5-fold CV intra-E102, même salle, sans OHE salle),
    et identifier les combinaisons minimales suffisantes.
  - Compléter l'ablation LORO (section 16duodecies) par une analyse dans les
    conditions opérationnelles où les modèles atteignent 90–93 %.

  Protocole :
  - Script : `scripts/feature_importance_e102.py`
  - Données : E102 complet (11 250 échantillons, 6 campagnes, 75 cellules)
  - Validation : StratifiedKFold 5-fold, random_state=42
  - Modèles : RF (100 arbres), KNN (k=7, poids distance), HistGB (lr=0.1, depth=8, iter=150)
  - Sortie : 11 combinaisons de features testées (de "Signal seul" à "Tout")

  Importances RF (Gini, entraîné sur tout E102) :

    signal_A3    28.7 %  ★ (la plus discriminante)
    signal_A2    28.1 %
    signal_A1    27.7 %
    Signal        15.0 %
    Noise          0.1 %  (quasi nulle)

  Résultats des sous-ensembles de features (accuracy 5-fold, arrondi à 0.1 %) :

  ┌─────────────────────────────────────────────────────────────────────────────────┐
  │  Sous-ensemble de features   │  n features │    RF    │    KNN   │   HistGB    │
  ├─────────────────────────────────────────────────────────────────────────────────┤
  │  Signal seul                 │      1      │   7.1 %  │   4.0 %  │    7.1 %    │
  │  Signal + Noise              │      2      │   7.3 %  │   4.2 %  │    7.1 %    │
  │  Signal + A1                 │      2      │  29.9 %  │  26.4 %  │   30.0 %    │
  │  Signal + A2                 │      2      │  28.6 %  │  25.9 %  │   28.6 %    │
  │  Signal + A3                 │      2      │  30.8 %  │  26.3 %  │   30.7 %    │
  │  Signal + A1 + A2            │      3      │  75.6 %  │  74.2 %  │   75.5 %    │
  │  Signal + A1 + A2 + A3       │      4      │  92.3 %  │  91.6 %  │   74.6 % !! │
  │  All 5 features (+ Noise)    │      5      │  92.3 %  │  91.6 %  │   74.6 % !! │
  │  No Noise (S + A1 + A2 + A3) │      4      │  92.3 %  │  91.6 %  │   74.6 % !! │
  │  No Signal (N + A1+A2+A3)    │      4      │  90.7 %  │  90.1 %  │   90.7 %    │
  │  Antennes seules (A1+A2+A3)  │      3      │  90.7 %  │  90.1 %  │   90.7 %    │
  └─────────────────────────────────────────────────────────────────────────────────┘

  Constats majeurs :

  1. La feature "Noise" est entièrement inutile (Signal+Noise = 7.3% vs Signal seul = 7.1%).
     Son importance RF (0.1 %) confirme qu'elle n'apporte aucune information discriminante.
     → Elle peut être supprimée sans perte pour tous les modèles testés.

  2. Les 3 signaux d'antenne (A1, A2, A3) sont les features dominantes :
     - A1+A2+A3 seuls (sans Signal ni Noise) : 90.7 %
     - Ajouter Signal : +1.6 pp (RF/KNN) ou −16 pp (HistGB) → voir point 3
     - Chaque antenne prise seule + Signal : ~27–31 % seulement

  3. Anomalie HistGB : "Signal" dégrade HistGB de 16 pp (90.7% → 74.6%).
     "Signal" est une agrégation des signaux antenne (corrélé avec A1/A2/A3).
     HistGB (gradient boosting) semble souffrir de cette redondance corrélée à
     max_iter=150 : l'espace de feature augmente sans information nouvelle, mais
     modifie le paysage d'optimisation. RF et KNN ne sont pas affectés.
     → Pour HistGB, la configuration optimale est [A1, A2, A3] (3 features, 90.7 %).
     → Pour RF et KNN, la configuration optimale est [Signal, A1, A2, A3] (92.3 / 91.6 %).

  4. Palier des 3 antennes → saut non-linéaire :
     - 2 antennes (Signal+A1+A2) : 75.6 % → 3 antennes : 92.3 % (+16.7 pp)
     - La troisième antenne (A3) apporte un gain majeur, non marginal.
     - Ce résultat justifie pleinement l'utilisation du routeur à 3 antennes.

  5. Cohérence avec l'ablation LORO (section 16duodecies) :
     En LORO, toutes les variations étaient < 0.5 pp car le plancher est ~1 %.
     En room-aware (où les modèles atteignent 90-93 %), les contributions réelles
     des features sont visibles. A3 à elle seule vaut +16.7 pp (RF), confirme
     que l'ablation LORO sous-estime l'importance des features.

  Recommandation pour l'article :
  - Feature set optimal pour la production : [Signal, A1, A2, A3] (4 features, sans Noise)
    pour RF et KNN. Pour HistGB : [A1, A2, A3] (3 features).
  - Minimum viable : [A1, A2, A3] (90.7 % RF/KNN/HistGB, si on veut le minimum unifié).
  - "Signal" et "Noise" : "Signal" apporte 1.6 pp pour RF/KNN, "Noise" est toujours inutile.

  Fichiers de résultats :
  - reports/benchmarks/feature_importance_e102.csv  — tableau complet 11 subsets × 3 modèles
  - reports/benchmarks/feature_importance_e102.json — importances Gini RF + résultats subsets

16quinquiesdecies. PROTOCOLE TRISCOPE — MODÈLE MULTI-TÂCHE UNIQUE (D005 + E101)
════════════════════════════════════════════════════════════════════════════════

  Objectif :
  - Évaluer la capacité d'un modèle unique (NN+L-KNN) à résoudre simultanément
    trois tâches de localisation à partir du seul vecteur RSSI brut :
      (1) Cellule fine (25 cm × 30 cm)
      (2) Distance routeur–tableau (2 m / 3 m / 4 m / 5 m)
      (3) Salle (D005 vs E101)

  Architecture TriScope :
  - Embedding NN+L-KNN (couches 48→24) entraîné sur la tâche cellule.
  - Deux têtes LogisticRegression ajoutées sur l'espace latent appris :
      * Head "distance" : classe parmi les distances mesurées (4 classes)
      * Head "salle"    : classe parmi D005 / E101 (2 classes)
  - Données : D005 (ddeuxmetres, dquatremetres) + E101 (dtroismetres, dcinqmetres)
    (E102 exclu car une seule salle/distance → tâche salle triviale)
  - Features : 5 RSSI bruts [Signal, Noise, A1, A2, A3]
  - Split : 80/20 stratifié (test_size=0.2)

  Protocole d'évaluation (robustesse statistique) :
  - 50 tirages aléatoires indépendants, 20 échantillons de test par tirage
  - Rapport moyenne ± écart-type sur les 50 répétitions

  Résultats (50 itérations × 20 échantillons = 1 000 prédictions consolidées) :

  ┌─────────────────────────────────────────────────────────────────────────┐
  │  Tâche                   │  Accuracy moy. │  Écart-type │  Min  │  Max  │
  ├─────────────────────────────────────────────────────────────────────────┤
  │  Cellule (25×30 cm)      │    97.60 %     │   ± 3.35 pp │  90 % │ 100 % │
  │  Distance routeur        │    57.10 %     │   ± 11.2 pp │  30 % │  85 % │
  │  Salle (D005 vs E101)    │    72.10 %     │   ± 11.0 pp │  50 % │  95 % │
  └─────────────────────────────────────────────────────────────────────────┘

  Résultat complémentaire (split fixe 80/20, n_test unique) :
  - Cellule : 96.32 %  |  Distance : 55.57 %  |  Salle : 73.02 %

  Lecture :
  - La tâche cellule est dominée par l'embedding KNN — les 97.6 % confirment
    que la représentation latente est suffisamment discriminante même dans un
    contexte multi-salle (D005 + E101 = 5 300 + 3 600 échantillons).
  - La tâche distance routeur (57 %) est difficile : les RSSI reflètent
    imparfaitement la distance absolue entre deux salles aux géométries
    différentes. La tête LogReg travaille sur un espace appris pour la cellule,
    pas pour la distance → dégradation attendue.
  - La tâche salle (72 %) est meilleure que random (50 %) mais reste modérée :
    D005 et E101 ont des profils RSSI globalement distincts, mais l'embedding
    normalisé pour la cellule estompe partiellement cette séparation.
  - L'écart-type élevé (±11 pp pour distance/salle) sur 20 échantillons
    reflète la variance d'échantillonnage, pas une instabilité du modèle.

  Enseignement pour l'article :
  - Un modèle NN+L-KNN entraîné pour la localisation cellule transfère
    naturellement une partie de la structure spatiale utile pour des tâches
    connexes (salle, distance), sans ré-entraînement.
  - Les performances cellule (97.6 %) en mode multi-salle avec tirage aléatoire
    sont légèrement supérieures au room-aware complet (94 %), car le tirage de
    20 échantillons favorise les cellules faciles.

  Fichiers de résultats :
  - reports/benchmarks/triscope_runs.json   — 50 tirages individuels + stats
  - reports/benchmarks/triscope_split.json  — résultat split fixe 80/20

Lecture synthétique :
  - La classe nominale 0.75 m concentre l'essentiel de la reconnaissance correcte.
  - Les classes extrêmes 0.00 m et 1.50 m sont majoritairement rabattues vers
    la classe intermédiaire 0.75 m.
  - Le signal RSSI contient donc une information de hauteur réelle, mais trop
    partielle pour fournir une séparation robuste des trois niveaux.

Conclusion scientifique :
  - L'ajout de la tête élévation n'apporte qu'un gain marginal sur la cellule.
  - Le signal RSSI porte une information de hauteur partielle, dominée ici par
    la classe nominale 0.75 m.
  - Dans l'état actuel de ce dataset, cette expérience ne suffit pas à elle
    seule pour démontrer que l'élévation est le facteur principal du shift ;
    elle suggère plutôt un effet couplé hauteur + orientation + contexte de
    campagne.
  - Pour renforcer cette thèse, il faudrait un protocole contrôlé où seule la
    hauteur varie, avec davantage de niveaux intermédiaires et/ou des features
    plus riches que 5 RSSI scalaires.

16octodecies. BENCHMARK AUXILIAIRE PYTORCH — NN+L-KNN
════════════════════════════════════════════════════════════════════════════════

Objectif :
  Vérifier la faisabilité d'une variante PyTorch du modèle NN+L-KNN, distincte
  du pipeline principal actuellement implémenté avec `sklearn MLPClassifier`.

Implémentation :
  - Script : `scripts/nn_lknn_torch.py`
  - Encoder : MLP PyTorch (5 entrées RSSI -> 64 -> 32 -> softmax)
  - Décodeur : KNN local sur l'embedding latent (k=5, poids distance)
  - Optimisation : Adam, lr=1e-3, weight_decay=1e-4
  - Prétraitement : `StandardScaler`
  - Environnement d'exécution : `.venv` locale, torch 2.10.0+cpu
  - Date d'exécution : 19 mars 2026

Protocole :
  - Split unique stratifié 75/25 (`test_size=0.25`)
  - Seed fixe : 42
  - CPU uniquement, threads bornés (`OMP_NUM_THREADS=1`)
  - Attention : ces runs sont auxiliaires et ne remplacent pas les tableaux
    principaux multi-seeds / room-aware du rapport.

Résultats auxiliaires obtenus :

  1) D005 (ddeuxmetres + dquatremetres, run par défaut du script)
     - train = 1 275 | test = 425
     - Accuracy cellule : 98.35 %
     - Erreur moyenne   : 0.0119 m

  2) E102 complet (exp1..exp6, distance 4 m)
     - train = 8 437 | test = 2 813
     - Accuracy cellule : 91.50 %
     - Erreur moyenne   : 0.1260 m

Lecture :
  - La variante PyTorch fonctionne correctement et retrouve un niveau de
    performance cohérent avec le pipeline `sklearn` existant.
  - Sur D005, le run rapide atteint 98.35 %, très proche des meilleurs résultats
    intra-salle déjà rapportés (~98 %).
  - Sur E102 complet, le run rapide atteint 91.5 %, légèrement sous les ~92.7 %
    du benchmark principal intra-salle, mais dans le même ordre de grandeur.
  - L'écart résiduel peut s'expliquer par le protocole plus simple ici
    (un seul split, 30 epochs, pas d'agrégation multi-seeds, pas de calibration
    ni de recherche d'hyperparamètres).

Conclusion pratique :
  - Oui, une version PyTorch de NN+L-KNN a été exécutée avec succès.
  - À ce stade, elle constitue une preuve de faisabilité technique crédible,
    mais pas encore un remplacement méthodologiquement strict du pipeline
    `sklearn` utilisé dans les sections comparatives principales.

16novemdecies. BENCHMARK AUXILIAIRE XGBOOST / RF — COLLECTE 360°
════════════════════════════════════════════════════════════════════════════════

Objectif :
  Tester la piste "collecte 360 degrés" : le routeur est orienté selon plusieurs
  directions afin d'exploiter les différences entre antennes A1/A2/A3 et
  d'estimer plus finement la position sur la grille.

Implémentation :
  - Script : `scripts/xgb_polar_360.py`
  - Environnement : `.venv` locale avec XGBoost disponible
  - Datasets du run final : E101 circulaire (`back`, `right`, `front`, `left`)
    + E102 orientations (`exp1` à `exp4`, sans campagnes élévation exp5/exp6)
  - Découpage temporel : chaque fichier de 25 mesures est découpé en fenêtres
    de 5 mesures, soit environ 1 seconde par fenêtre
  - Taille obtenue : 2 940 fenêtres, split stratifié 90/10, 294 fenêtres test
  - Important : les variables de contexte `orientation_sin_input` /
    `orientation_cos_input` ont été exclues des features modèle afin de ne pas
    donner au modèle le label d'orientation de la campagne.

Features construites :
  - Statistiques par signal : moyenne, écart-type, médiane, minimum, maximum
  - Transformations de puissance : `log1p(abs(RSSI))`, conversion dBm -> mW
  - Différences inter-antennes : A1-A2, A1-A3, A2-A3
  - Différences signal global vs antennes : Signal-A1/A2/A3
  - Valeurs absolues et carrés des différences pour accentuer les asymétries
  - Ratios de puissance entre antennes en échelle mW

Modèles évalués :
  1) `xgb_polar` :
     trois têtes de régression XGBoost séparées :
       - rayon / distance polaire à l'origine de la grille
       - sin(theta)
       - cos(theta)

  2) `rf_polar` :
     même formulation polaire, mais avec RandomForestRegressor.

  3) `xgb_cell_classifier` et `rf_cell_classifier` :
     classification directe de la cellule, puis projection au centre de la
     cellule prédite.

  4) `weighted_xgb_rf` :
     moyenne pondérée 70 % XGBoost / 30 % RF sur les sorties polaires.

  5) `arbitrated_proxy` :
     arbitrage XGB/RF avec lissage fort quand les deux modèles divergent.
     Ce mode est seulement un proxy, car le dataset n'est pas une vraie
     trajectoire temps réel ordonnée.

  - XGBoost et RandomForest sont très proches en accuracy cellule ; XGBoost

    accuracy légèrement plus élevée.
  - La formulation polaire reste intéressante conceptuellement, mais elle perd
    de l'information discrète utile : prédire un rayon et un angle continus ne
    suffit pas à retrouver aussi bien la cellule exacte.
  - Le lissage/arbitrage de type "simili-Kalman" dégrade ici les métriques, car
    il est appliqué à des fenêtres triées par campagne/cellule et non à une
    vraie trajectoire physique. Il doit être re-testé sur flux temps réel.

  - RF cellule      : erreur latérale moyenne 0.326 m, profondeur 0.098 m

    contrairement à l'hypothèse initiale attendue pour certains contextes
    indoor. Ce point peut venir du mélange E101+E102 et du repère grille utilisé.

18. DEEP LEARNING AVANCÉ 2026 — PANORAMA DES APPROCHES RÉCENTES
════════════════════════════════════════════════════════════════════════════════

  Ce chapitre documente 14 approches algorithmiques testées ou implémentées
  entre mars et avril 2026, couvrant les grandes familles identifiées dans la
  littérature récente en fingerprinting RSSI et machine learning tabulaire.
  Les benchmarks d'avril 2026 ont été consolidés avec un artefact JSON ou un
  statut d'exécution explicite.

  Référence de base (rappel) :
    KNN (k=7, poids distance) room-aware : 91.5–91.8 %  |  err. 0.097–0.103 m
    KNN intra-salle E102 (80/20)          : 92.4–92.7 %  |  err. 0.112–0.114 m
    LORO tous modèles                     : ~1–2 %         → non transférable

────────────────────────────────────────────────────────────────────────────────
18.1 TabNet — Attention séquentielle sur données tabulaires
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  TabNet (Arik & Pfister, Google Brain, AAAI 2021) introduit un mécanisme
  d'attention par étapes pour sélectionner séquentiellement les features
  pertinentes. Très efficace sur les benchmarks tabulaires standards (100+
  features hétérogènes).

  Implémentation : script `scripts/tabnet_benchmark.py` (pytorch-tabnet 4.1.0)
  Protocoles : room-aware (80/20), intra-salle E102 (80/20)

  Résultats (TabNet vs KNN vs RF) :

  ┌────────────────────────┬──────────────┬──────────────┬───────────────┐
  │ Protocole              │ KNN          │ RF           │ TabNet        │
  ├────────────────────────┼──────────────┼──────────────┼───────────────┤
  │ Room-aware (n=29 730)  │ 91.7 % 0.103m│ 91.8 % 0.101m│ 25.4 % 0.989m │
  │ E102 intra-room        │ 92.4 % 0.112m│ 93.3 % 0.105m│ 51.6 % 0.700m │
  └────────────────────────┴──────────────┴──────────────┴───────────────┘

  Analyse :
  - TabNet échoue significativement (25 % room-aware, 52 % E102) par rapport
    aux baselines KNN/RF (91–93 %).
  - Explication structurelle : le mécanisme d'attention de TabNet est conçu
    pour sélectionner parmi de nombreuses features hétérogènes. Avec seulement
    5 features RSSI d'importance quasi-égale (~27–29 % chacune selon Gini RF),
    l'attention séquentielle n'a rien à sélectionner de façon discriminante.
  - TabNet requiert typiquement 10× à 100× plus de données que KNN pour
    converger sur des problèmes à faible dimensionnalité.

  Conclusion : TabNet inadapté pour ce problème 5D à forte symétrie de features.

────────────────────────────────────────────────────────────────────────────────
18.2 Adaptation de domaine — CORAL + DANN
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  L'échec LORO (~1–2 %) s'explique par un shift de distribution inter-salles.
  Deux approches classiques d'adaptation de domaine ont été testées :
  - CORAL (Correlation ALignment) : aligne la matrice de covariance de la
    source sur celle de la cible (Sun & Saenko, 2016).
  - DANN (Domain Adversarial Neural Network) : entraîne un encodeur à
    produire des représentations indépendantes du domaine via un gradient
    reversal layer (Ganin et al., JMLR 2016).

  Protocole : LORO inter-salle (4 salles, hold-one-room-out).
  Script : `scripts/domain_adaptation_loro.py`

  Résultats moyens LORO (4 folds) :

  ┌──────────────────┬───────────────┬─────────────┐
  │ Méthode          │ Acc. cellule  │ Err. moy.   │
  ├──────────────────┼───────────────┼─────────────┤
  │ KNN_vanilla      │ 1.16 % ±0.82  │ 1.375 m     │
  │ RF_vanilla       │ 1.13 % ±0.52  │ 1.355 m     │
  │ CORAL + KNN      │ 1.25 % ±0.64  │ 1.345 m     │
  │ CORAL + RF       │ 0.80 %        │ 1.358 m     │
  │ DANN             │ 1.32 % ±0.33  │ 1.349 m     │
  └──────────────────┴───────────────┴─────────────┘

  Analyse :
  - CORAL apporte un gain marginal (+0.09 pp) sur KNN, mais dégrade RF.
  - DANN (+0.16 pp) est le seul modèle qui réduit la variance (±0.33 vs ±0.82),
    signe que l'alignissement de domaine apporte de la stabilité mais pas de
    précision.
  - Le plafond LORO (~1–2 %) reflète un shift structurel RSSI (géométrie
    différente par salle, gains d'antenne différents) que les méthodes
    d'alignissement statistique ne peuvent pas compenser.

  Résultat notable pour l'article : résultat négatif solide. CORAL et DANN
  confirment l'irréductibilité du shift inter-salle sur ce type de fingerprint.

────────────────────────────────────────────────────────────────────────────────
18.3 SAE + Multi-label — Stacked Autoencoder avec classification row/col
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Inspiré de Nabati et al. (Nature/Sensors 2025). Décompose le problème de
  classification de cellule (75+ classes) en deux sous-problèmes indépendants :
  classification de ligne (row) + classification de colonne (col).
  Un Stacked Autoencoder (SAE) pré-entraîne une représentation compacte par
  reconstruction RSSI avant fine-tuning sur les deux têtes.

  Architecture :
    SAE encoder : RSSI → hidden(32) → bottleneck(16) → [row_head, col_head]
    Decoder : bottleneck → reconstruction RSSI (loss MSE)
    Loss totale : L_row + L_col + 0.3 × L_reconstruction
    Inférence : score(cell) = P(row=r) × P(col=c) → argmax sur les cellules valides

  Protocoles exécutés : room-aware, E102 (`--skip-loro`).
  Script : `scripts/sae_multilabel_benchmark.py`
  Commande : `--epochs 100 --hidden 32 --bottleneck 16 --recon-weight 0.3 --skip-loro`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :
  - Room-aware :
    * KNN : 91.69 %, 0.103 m
    * SAE+MultiLabel : 4.91 %, 1.276 m
    * SAE+KNN : 91.54 %, 0.103 m
  - E102 :
    * KNN : 92.44 %, 0.112 m
    * SAE+MultiLabel : 5.51 %, 1.279 m
    * SAE+KNN : 92.76 %, 0.108 m

  Lecture :
  - La décomposition row/col directe ne fonctionne pas : elle casse la structure
    de classe fine et accumule les erreurs de ligne/colonne.
  - L'encodeur SAE n'améliore pas le plafond room-aware, mais l'espace latent
    utilisé par KNN conserve la performance et donne un léger gain sur E102.
  - À garder comme prétraitement léger possible, pas comme classifieur autonome.

────────────────────────────────────────────────────────────────────────────────
18.4 SSGAN — Semi-Supervised GAN pour augmentation de données
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Contrairement à un CGAN classique (données labellisées seulement), le SSGAN
  utilise aussi les données non-labellisées dans un discriminateur à double
  tête (real/fake + classification cellule), guidant mieux le générateur.
  Référence : Odena (2016) ; travaux Wi-Fi fingerprint SSGAN (Sensors/MDPI 2024).

  Architecture :
    Générateur : z(32) + label_emb(16) → hidden(128) → RSSI synthétique
    Discriminateur :
      - Tête real/fake (BCE loss)
      - Tête classification cellule (CE loss, sur données labellisées seulement)
    λ_cls = 1.0 (poids de la perte de classification dans D)

  Protocole exécuté : E102 LOCO campagne, n_synth=10 samples/cellule.
  Script : `scripts/ssgan_augmentation.py`
  Commande : `--rooms E102 --epochs 100 --hidden 64 --batch-size 128 --n-synth 10`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultat moyen E102 :
  - KNN baseline LOCO campagne : 1.91 %
  - KNN + SSGAN augmentation : 1.82 %
  - Delta : -0.09 pp

  Lecture : l'augmentation SSGAN ne réduit pas le shift de campagne. Les
  fingerprints synthétiques ne créent pas d'information utile pour KNN et
  peuvent même ajouter du bruit.

────────────────────────────────────────────────────────────────────────────────
18.5 TabDDPM — Diffusion probabiliste pour augmentation tabulaire
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  TabDDPM (Kotelnikov et al., ICML 2023) adapte les modèles de diffusion
  débruisants (DDPM) aux données tabulaires. Conditionné sur (cellule, campagne),
  il génère des fingerprints RSSI synthétiques pour les conditions sous-représentées.

  Architecture prévue :
    Schedule : cosine noise schedule (Nichol & Dhariwal, 2021), T=100 pas
    Débruiteur : MLP conditionné temps + label (time_emb + label_emb + MLP profond)
    Inférence DDPM : processus inverse stochastique sur T pas

  Protocole exécuté : E102 LOCO campagne, version ultra-courte T=10,
  5 epochs, n_synth=2, pour borner le coût CPU.
  Script : `scripts/tabddpm_augmentation.py`
  Statut : [TERMINÉ — run court du 26 avril 2026]

  Résultat moyen E102 :
  - KNN baseline LOCO campagne : 1.91 %
  - KNN + TabDDPM augmentation : 1.79 %
  - RF + TabDDPM augmentation : 1.70 %

  Lecture : le run court ne montre aucun gain. Un run long DDPM peut encore être
  relancé pour analyse, mais la piste n'est pas prioritaire tant que le split
  LOCO campagne reste au plancher.

────────────────────────────────────────────────────────────────────────────────
18.6 TabM + KAN — Ensemble de mini-MLP et Kolmogorov-Arnold Networks
────────────────────────────────────────────────────────────────────────────────

  TabM (NeurIPS 2024) :
  Ensemble de k=16 mini-MLP indépendants (même architecture, poids distincts),
  logits moyennés à la sortie. L'idée centrale : la diversité des initialisations
  suffit à construire un ensemble efficace sans bagging.
    Architecture : k×(Linear→BN→GELU→Dropout)×2 + linear → mean pooling
    Protocoles : room-aware, E102, LORO. Script : `scripts/tabm_kan_benchmark.py`

  KAN (Kolmogorov-Arnold Networks, Liu et al., 2024) :
  Remplace les neurones linéaires par des fonctions splines B-splines apprenables
  sur les arêtes. Potentiellement plus interprétable et plus précis sur les données
  à structure régulière.
    Architecture : KAN(width=[n_features, hidden, n_classes], grid=5, k=3)
    Limite : KAN est coûteux en mémoire pour n_classes > 80, skip si dépassé.

  Script : `scripts/tabm_kan_benchmark.py`
  Commande : `--tabm-k 8 --tabm-hidden 64 --tabm-epochs 100 --skip-kan --skip-loro`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :
  - Room-aware :
    * KNN : 91.69 %, 0.103 m
    * RF : 91.81 %, 0.101 m
    * TabM : 38.08 %, 0.872 m
  - E102 :
    * KNN : 92.44 %, 0.112 m
    * RF : 93.33 %, 0.105 m
    * TabM : 33.07 %, 0.986 m

  Lecture : TabM reste très inférieur aux méthodes locales. KAN n'a pas été
  exécuté dans ce run (`--skip-kan`) ; sur ce problème à nombreuses classes,
  le script le limite aussi lorsque n_classes dépasse le seuil mémoire.

────────────────────────────────────────────────────────────────────────────────
18.7 ProtoNet + Calibration few-shot cible
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Les ProtoNets (Snell et al., NeurIPS 2017) apprennent un espace métrique
  où chaque classe est représentée par le centroïde de ses exemples de support.
  Pertinent pour le fingerprinting : à la mise en service d'une nouvelle pièce,
  K mesures par cellule suffisent comme calibration.

  Architecture :
    ProtoEncoder : Linear→BN→ReLU×2 → emb_dim(32)
    Entraînement : 2000 épisodes épisodiques (n_way=20, n_support+n_query)
    Protocole LORO : K={1, 3, 5, 10, 25} supports/cellule, 5 tirages par K

  Scripts :
  - `scripts/protonet_fewshot.py`
  - `scripts/fewshot_target_calibration.py`
  - `scripts/fewshot_campaign_calibration.py`
  - `scripts/targetknn_stress_test.py`

  Commandes principales :
  - ProtoNet S=5 :
    `--episodes 2000 --n-way 20 --n-support 5 --n-query 10 --n-trials 5`
  - ProtoNet S=1 :
    `--episodes 2000 --n-way 20 --n-support 1 --n-query 15 --n-trials 5`
  - Calibration cible :
    `--k-values 1 2 3 5 10 15 20 25 --n-trials 20 --knn-neighbors 3`
  - Calibration campagne LOCO :
    `--rooms B121 D005 E101 E102 --k-values 1 2 3 5 10 15 20 25 --n-trials 20`

  Statut : [TERMINÉ — runs du 25–26 avril 2026]

    * B121 : 87.52 %

    * E101 : 87.85 %

  Intérêt pour l'article :
  - Quantifie le « coût de recalibration » : combien de mesures par cellule
    suffisent pour atteindre un niveau d'accuracy acceptable dans une nouvelle salle.
  - Complémentaire de MAML (section 18.15) : ProtoNet n'adapte pas les poids,
    seulement les prototypes.
  - Résultat principal : le few-shot labellisé change vraiment le régime LORO,
    mais le meilleur outil n'est pas ProtoNet ; c'est un KNN local entraîné sur
    quelques mesures de la salle cible.
  - ProtoNet reste intéressant en K=1, mais il n'exploite pas correctement
    l'augmentation de K. TargetKNN, lui, suit une courbe monotone et exploitable.
  - En LOCO, le même principe est encore plus fort : quelques mesures de la
    campagne cible suffisent à revenir proche du plafond room-aware.

  Stress test anti-fuite TargetKNN :

  Objectif :
  Vérifier que le gain TargetKNN n'est pas une erreur de protocole, ni une fuite
  de données où les points de test auraient déjà été vus à l'entraînement.

  Commandes exécutées :
  - LORO strict :
    `scripts/targetknn_stress_test.py --k-values 1 5 10 25 --nn-values 1 3 5 7 --n-trials 20 --noise-sigmas 0 1 3 --hybrid-k-values 5 10 25 --control-k-values 5 10 25 --output-name targetknn_stress_test_loro_audit.json`
  - LOCO strict :
    `scripts/fewshot_campaign_calibration.py --rooms B121 D005 E101 E102 --k-values 5 10 25 --n-trials 10 --output-name targetknn_loco_audit.json`

  Garde-fous du protocole :
  - En LORO, la salle cible est totalement absente de l'entraînement source.
    Le StandardScaler est ajusté uniquement sur les salles sources.
  - En LOCO, la campagne cible est totalement absente de l'entraînement source.
  - Les K supports labellisés sont tirés dans la cible, puis retirés du set de
    requête. Les samples de requête ne sont donc jamais utilisés pour entraîner
    TargetKNN.
  - Le JSON d'audit vérifie à chaque tirage que `support_query_overlap_max = 0`.
  - Contrôle négatif : on relance TargetKNN avec les mêmes supports, mais avec
    les labels de support mélangés. Si le score venait d'une fuite ou d'une
    mémorisation cachée, il resterait élevé. S'il vient bien des quelques mesures
    labellisées, il doit retomber vers le hasard.

  Résultats LORO strict, moyenne 4 salles :
  - SourceKNN sans label cible : 1.30 %
  - TargetKNN K=1  : 18.12 %, 1.047 m
  - TargetKNN K=5  : 49.49 %, 0.653 m
  - TargetKNN K=10 : 66.91 %, 0.429 m
  - TargetKNN K=25 : 84.65 %, 0.196 m
  - Contrôle labels mélangés :
    * K=5  : 1.53 %
    * K=10 : 1.77 %
    * K=25 : 1.71 %
  - Audit : `target_room_present_in_source_training = false` pour les 4 folds ;
    `support_query_overlap_max = 0` pour les 4 folds.

  Résultats LOCO strict, moyenne B121/D005/E101/E102 :
  - KNN source sans label de campagne cible : 1.66 %
  - TargetKNN K=5  : 89.80 %, 0.117 m
  - TargetKNN K=10 : 94.57 %, 0.058 m
  - TargetKNN K=25 : 96.25 %, 0.039 m
  - Contrôle labels mélangés :
    * K=5  : 2.00 %
    * K=10 : 1.73 %
    * K=25 : 1.63 %
  - Audit : `held_out_campaign_present_in_source_training = false` sur tous
    les folds ; `support_query_overlap_max = 0` sur tous les folds.

  Lecture :
  - Le gain ne survit pas quand les labels de support sont mélangés : il retombe
    au niveau du hasard. Cela valide que TargetKNN n'utilise pas une fuite de
    données, mais bien l'information contenue dans quelques mesures labellisées
    de la cible.
  - Le benchmark hybride Source+Target est moins bon que TargetKNN pur en LORO
    (K=25 : 68.32 % vs 84.55 %). Les anciennes salles ajoutent donc du bruit
    plus qu'elles n'aident : le signal utile vient de la calibration locale.
  - Ajouter du bruit artificiel aux supports LORO confirme que ce sont les vraies
    mesures de calibration qui portent le résultat. A K=10, TargetKNN passe de
    66.74 % sans bruit à 7.25 % avec sigma=1 dBm et 3.54 % avec sigma=3 dBm.
    En pratique, les supports doivent donc être collectés proprement plutôt que
    simulés.

18.8 TTA / Tent — Adaptation test-time par minimisation d'entropie
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Test-Time Adaptation (TTA) adapte le modèle pendant l'inférence, sans accès
  aux labels. Tent (Wang et al., ICLR 2021) minimise l'entropie des prédictions
  en ajustant uniquement les paramètres affines des couches BatchNorm.

  Modes testés :
    vanilla    : MLP figé, pas d'adaptation
    bn_stats   : mise à jour des statistiques BN avec les données test (pas de grad)
    tent       : minimisation entropie sur γ, β BN (1 epoch, n_steps=30)
    full_ent   : minimisation entropie sur tous les paramètres

  Protocole exécuté : LORO 4 folds, 100 epochs MLP, 10 tent steps.
  Script : `scripts/tta_localization.py`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats moyens LORO :
  - vanilla : 1.71 % ± 1.03 pp
  - bn_stats : 1.71 % ± 1.03 pp
  - tent : 1.43 % ± 0.19 pp
  - full_ent : 1.38 % ± 0.39 pp

  Hypothèse : TTA/Tent peut compenser partiellement le shift inter-salle en
  ajustant les statistiques BN sur le batch test de la salle tenue hors.
  Résultat : hypothèse infirmée. Sans labels cible, l'adaptation par entropie
  renforce parfois des prédictions confiantes mais fausses.

18.9 Metric Learning — SupCon, Triplet Loss, Proxy-NCA
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Au lieu d'entraîner un embedder avec la loss CE classique, le metric learning
  entraîne explicitement l'espace latent pour que les vecteurs de la même cellule
  soient proches et les vecteurs de cellules différentes éloignés.

  Trois méthodes testées :
  1. SupCon (Khosla et al., NeurIPS 2020) :
       sim(z_i, z_j) = z_i·z_j/τ  ;  L = –log(Σ_pos exp(sim) / Σ_all exp(sim))
       Température τ=0.07. La plus stable des trois.

  2. Triplet Margin Loss (Hermans et al., 2017) :
       L = ReLU(d(anchor, pos_hard) – d(anchor, neg_hard) + margin)
       Hard-mining batch : hardest positive + hardest negative.

  3. Proxy-NCA :
       Un proxy appris par classe (embeding centroïde). CE sur les similarités
       cosinus entre points et proxies. Passe à l'échelle pour 100+ classes.

  Architecture commune : MetricEncoder(Linear→BN→ReLU)×2 → emb_dim(32),
  sortie L2-normalisée. Évaluation : KNN dans l'espace d'embedding.

  Protocoles exécutés : room-aware, E102 (`--skip-loro`).
  Script : `scripts/metric_learning_benchmark.py`
  Commande : `--methods supcon triplet proxy --epochs 100 --hidden 64 --emb-dim 32 --skip-loro`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :
  - Room-aware : KNN raw 91.69 %, SupCon 91.59 %, Triplet 91.47 %, Proxy 91.59 %
  - E102 : KNN raw 92.44 %, SupCon 92.80 %, Triplet 92.80 %, Proxy 92.76 %

  Lecture : l'embedding métrique ne dépasse pas le KNN brut en multi-room, mais
  il apporte un petit gain intra-E102. C'est une piste d'optimisation fine, pas
  un changement de paradigme.

18.10 DeepSets — Apprentissage sur ensembles de scans RSSI
────────────────────────────────────────────────────────────────────────────────

  Contexte (Zaheer et al., NeurIPS 2017) :
  Chaque position physique est mesurée 25 fois. Les méthodes classiques traitent
  chaque ligne indépendamment. DeepSets apprend f(X_set) = ρ(Σ_i φ(x_i)) :
  une représentation invariante à la permutation de l'ensemble de mesures.

  Architecture :
    φ (per-element) : Linear(n_feat→64) → ReLU → emb(64)
    Agrégation : mean ou max sur l'ensemble de scans
    ρ (set-level) : Linear(64→128) → BN → ReLU → Linear(64) → classif

  Construction des ensembles : groupe par (grid_cell, campaign), échantillonne
  des sous-ensembles de taille set_size=5 dans le run exécuté. Le split train/test se fait au
  niveau des groupes (anti-leakage) et non des lignes individuelles.

  Deux modes évalués :
    DeepSets_mean / DeepSets_max : classification directe de l'ensemble
    KNN+DeepSets : KNN dans l'espace d'embedding ρ(·)

  Protocoles : room-aware (groupes 80/20), E102.
  Script : `scripts/deepsets_benchmark.py`
  Commande : `--aggregations mean max --set-size 5 --epochs 100`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :
  - Room-aware strict group split :
    * KNN_instance : 0.84 %, 1.273 m
    * DeepSets_mean : 1.26 %, 1.258 m
    * DeepSets_max : 1.26 %, 1.330 m
  - E102 strict group split :
    * KNN_instance : 3.29 %, 1.341 m
    * DeepSets_mean : 0.89 %, 1.330 m
    * KNN+DeepSets_max : 1.78 %, 1.275 m

  Intérêt : si le modèle apprend la variabilité intra-cellule (et pas seulement
  la moyenne), il devrait être plus robuste aux perturbations RSSI.
  Lecture : le protocole par groupes est beaucoup plus dur que le 80/20 ligne
  à ligne, donc ces scores ne sont pas directement comparables à KNN 92 %. Dans
  ce cadre anti-leakage, DeepSets n'apporte pas encore de robustesse.

18.11 GNN — Réseau de Neurones sur Graphe de Cellules
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Les cellules forment une grille spatiale régulière : les adjacences physiques
  sont connues. Un GNN peut imposer une cohérence spatiale pendant l'apprentissage
  (les cellules voisines doivent avoir des représentations proches), ce qui est
  plus fort scientifiquement qu'un post-traitement de lissage.

  Architecture :
    Graphe : A_hat = D^{-1/2}(A+I)D^{-1/2}, arêtes entre cellules à ≤0.6 m
    Nœuds : prototypes RSSI par cellule (moyenne des embeddings d'entraînement)
    GCN : 2 couches GCNLayer (message passing = A_hat @ H @ W + b), ReLU
    Query encoder : MLP(RSSI → hidden → emb_dim) pour mapper les requêtes
    Classification : similarité cosinus requête vs nœuds GNN → argmax

  Entraînement joint (deux objectifs) :
    L_node : chaque nœud doit prédire sa propre classe (auto-supervision)
    L_inst : chaque instance d'entraînement doit être proche de son nœud GNN

  Protocoles : room-aware (80/20), E102.
  Script : `scripts/gnn_localization.py`
  Correctif appliqué : conversion explicite des tenseurs GNN en `float32`.
  Commande : `--epochs 100 --hidden 64 --emb-dim 64 --n-layers 2`
  Statut : [TERMINÉ — run du 26 avril 2026]

  Résultats :
  - Room-aware : KNN 91.52 %, GNN_cell 2.59 %, 1.256 m
  - E102 : KNN 92.44 %, GNN_cell 2.84 %, 1.289 m

  Lecture : le GNN de prototypes n'apprend pas un espace requête→cellule
  exploitable. Il faut soit revoir l'objectif contrastif, soit utiliser le graphe
  comme régularisation d'un classifieur fort plutôt qu'un modèle autonome.

18.12 Self-Supervised Pretraining — MAE, DAE, SimCLR
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Le pretraining auto-supervisé exploite les données non-labellisées pour
  apprendre une représentation générique avant fine-tuning sur les labels.
  Particulièrement pertinent si les données de calibration sont rares.

  Trois objectives SSL :
  1. MAE (Masking Autoencoder, He et al., 2022) :
       Masque p=40 % des features → 0, reconstruit les originaux (loss MSE).
       Force l'encodeur à apprendre les corrélations inter-antennes.

  2. DAE (Denoising Autoencoder) :
       Ajoute N(0, σ=0.3) au signal, reconstruit la version propre.
       Plus doux que le masquage, adapté aux signaux continus.

  3. SimCLR (Chen et al., 2020) :
       Deux vues augmentées (bruit indépendant) du même scan → NT-Xent loss.
       Apprend une représentation invariante au bruit de mesure.

  Pipeline :
    Phase 1 (pretraining) : 150 epochs, toutes les données d'entraînement.
    Phase 2 (fine-tuning)  : 150 epochs, labels cellule, lr×0.1.
    Évaluation : accuracy directe + KNN dans l'espace fine-tuné.

  Script : `scripts/selfsup_pretraining.py`
  Commande : `--methods mae dae simclr --pretrain-epochs 60 --finetune-epochs 80`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :
  - Room-aware :
    * KNN : 91.52 %, 0.102 m
    * MLP scratch : 54.78 %, 0.605 m
    * MAE/DAE/SimCLR finetune : 10–13 %, ~1.19 m
    * MAE/DAE/SimCLR + KNN : 91.56–91.57 %, ~0.102–0.103 m
  - E102 :
    * KNN : 92.44 %, 0.114 m
    * MLP scratch : 61.20 %, 0.587 m
    * MAE/DAE/SimCLR + KNN : 92.80 %, 0.105–0.108 m

  Lecture : le pré-entraînement seul ne suffit pas pour fine-tuner un classifieur
  fiable, mais l'espace appris reste compatible avec KNN et apporte un petit gain
  sur E102. La bonne combinaison est donc "embedding + méthode locale".

18.13 Spatially-Aware Loss + Localisation Hiérarchique
────────────────────────────────────────────────────────────────────────────────

  (a) Spatially-aware loss :

  Motivation : la CE standard pénalise identiquement une erreur d'une cellule
  (≈0.28 m) et une erreur de 10 cellules (2 m). Ceci crée un désalignement
  entre la loss d'entraînement et la métrique terrain mean_error_m.

  Formulation :
    L_spatial = α × CE + (1-α) × E[Σ_j P(j|x) × dist(cell_j, cell_true)]
  Avec α=0.5 par défaut. La pénalité spatiale est différentiable (softmax).

  (b) Localisation hiérarchique :

  Motivation : partitionner l'espace en zones (1.5 m × 1.5 m), prédire d'abord
  la zone puis restreindre la recherche de cellule à la zone prédite.
  Les erreurs catastrophiques (sauts à l'autre bout de la pièce) sont ainsi
  bloquées par le classifier de zone.

  Architecture :
    Classificateur de zone : MLP(64) → n_zones (auto-calculé sur les coordonnées)
    Classificateur cellule : MLP(128), même entraînement que MLP_CE
    Inférence : pred_zone → mask logits cellule → argmax sur cellules de la zone

  Protocoles : room-aware, E102 (+ LORO optionnel).
  Script : `scripts/spatial_loss_benchmark.py`
  Correctif appliqué : `hierarchical_predict` utilise maintenant `torch.no_grad()`
  pour convertir les logits en NumPy.
  Commande : `--epochs 100 --hidden 64 --batch-size 512 --alpha 0.5`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :
  - Room-aware :
    * KNN : 91.52 %, 0.102 m
    * MLP_CE : 35.39 %, 0.870 m
    * MLP_SpatialCE : 33.64 %, 0.796 m
    * Hierarchical : 35.99 %, 0.772 m
  - E102 :
    * KNN : 92.44 %, 0.114 m
    * MLP_CE : 44.13 %, 0.776 m
    * MLP_SpatialCE : 40.44 %, 0.722 m
    * Hierarchical : 47.47 %, 0.622 m

  Lecture : la loss spatiale réduit l'erreur moyenne mais baisse l'accuracy
  cellule. Le modèle hiérarchique est le meilleur des variantes MLP, surtout
  sur E102, mais reste très inférieur à KNN.

18.14 Mixture-of-Experts (MoE) — Spécialisation par régime radio
────────────────────────────────────────────────════════════════════════════════

  Contexte :
  Hypothèse : les fingerprints RSSI se décomposent en "régimes radio" latents
  (orientation du routeur, position dans la salle, effet de multipath),
  qu'un modèle unique apprend à moyenner. MoE apprend des experts spécialisés
  par régime + un gate léger.

  Architecture (Shazeer et al., ICLR 2017) :
    Gate : MLP(n_feat → 32 → E) → softmax, top-k=2 routing
    K experts : chacun un MLP(n_feat → 64 → 64 → n_classes)
    Output : moyenne pondérée par les poids gate des k experts activés
    Load balancing loss : MSE(mean(gate_probs), 1/E) × λ=0.01

  Expert counts testés : 2, 4, 8 experts.
  Protocoles : room-aware, E102 (+ LORO optionnel).
  Script : `scripts/moe_localization.py`
  Commande : `--n-experts 2 4 --top-k 2 --epochs 100 --hidden 64`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :
  - Room-aware :
    * KNN : 91.52 %, 0.102 m
    * MLP : 23.73 %, 1.035 m
    * MoE 2 experts : 33.96 %, 0.897 m
    * MoE 4 experts : 46.45 %, 0.730 m
  - E102 :
    * KNN : 92.44 %, 0.114 m
    * MLP : 26.00 %, 1.107 m
    * MoE 2 experts : 36.13 %, 0.902 m
    * MoE 4 experts : 50.44 %, 0.707 m

  Lecture : le routage par experts améliore clairement un MLP global, ce qui
  confirme une hétérogénéité des régimes radio. Mais même MoE-4 reste loin du
  comportement local de KNN.

18.15 FOMAML — Meta-apprentissage pour adaptation rapide à une nouvelle salle
────────────────────────────────────────────────────────────────────────────────

  Contexte (Finn et al., ICML 2017) :
  MAML apprend une initialisation θ* telle que K étapes de gradient sur une
  nouvelle tâche (salle inconnue) suffisent à atteindre une bonne accuracy.
  Contrairement à ProtoNet (section 18.7), MAML adapte tous les paramètres.

  Implémentation : First-Order MAML (FOMAML) pour éviter le calcul de la
  hessienne. L'approximation est proche de MAML standard sur la plupart des tâches.

  Paramètres :
    Inner loop : SGD, lr=0.05, n_inner_steps=5 (train) / 10 (test)
    Meta-optimizer : Adam, meta_lr=1e-3, 8 tâches/epoch, 100 epochs
    Architecture : MAMLClassifier(n_feat → 64 → 64 → n_classes)

  Protocole LORO : meta-entraînement sur N-1 salles, meta-test sur la salle
  tenue hors avec K={1, 5, 10, 25} exemples de support par cellule.

  Script : `scripts/maml_localization.py`
  Commande : `--meta-epochs 50 --tasks-per-epoch 4 --n-support 5 --n-query 10 --k-shots 1 5 10 25`
  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats LORO moyens :
  - K=1  : 1.59 %
  - K=5  : 1.84 %
  - K=10 : 1.78 %
  - K=25 : 1.88 %

  Comparaison avec ProtoNet :
  - ProtoNet : centroïdes, pas d'adaptation des poids
  - FOMAML   : adaptation de tous les poids en K gradient steps
  - Ensemble des deux : couvre le spectre "adaptation-légère" vs "adaptation-lourde"
  - Résultat : FOMAML ne sort pas du plancher LORO. ProtoNet est nettement plus
    prometteur dans ce jeu de données, probablement parce que l'adaptation de
    poids est trop instable avec peu de salles source.

18.16 Conformal Prediction + Classification Sélective
────────────────────────────────────────────────────────────────────────────────

  Contexte :
  Face à l'échec LORO, une alternative crédible est un système qui s'abstient
  plutôt que de faire une prédiction erronée, avec garantie de couverture
  probabiliste. Deux mécanismes complémentaires :

  (a) Split Conformal Prediction (Angelopoulos & Bates, 2023) :
  Utilise un jeu de calibration (20 % du train) pour calculer des scores de
  non-conformité s_i = 1 – P(y_true | x_i) sur les données calibration.
  Seuil τ_α = Quantile_{(1-α)(1+1/n)} des scores.
  En test, le jeu de prédiction = {cellules j : 1–P(j|x) ≤ τ_α}.
  Garantie de couverture marginale ≥ 1–α sans hypothèse distributionnelle.

  Métriques conformal :
    - Couverture marginale (doit être ≥ 1–α)
    - Taille moyenne du jeu de prédiction
    - Distance minimale au vrai point dans le jeu (erreur spatiale optimiste)

  Niveaux α testés : 5 %, 10 %, 20 %.

  (b) Classification sélective (abstain-or-predict) :
  Seuil de confiance τ ∈ [0.05, 0.99] sur max-softmax.
  Pour chaque τ : calcule (coverage_rate, accuracy_when_answered).
  Trace la frontière coverage-accuracy, permettant d'arbitrer entre couverture
  et précision garantie.

  Protocoles : room-aware (80/20 + split calibration 20 %), E102.
  Script : `scripts/conformal_localization.py`
  Commande :
    OMP_NUM_THREADS=1 MKL_NUM_THREADS=1 OPENBLAS_NUM_THREADS=1
    NUMEXPR_NUM_THREADS=1 PYTHONPATH=src ./.venv/bin/python
    scripts/conformal_localization.py --epochs 100 --hidden 64
    --alphas 0.05 0.10 0.20

  Statut : [TERMINÉ — run du 25 avril 2026]

  Résultats :

  ┌───────────────┬──────────────┬──────────────┬──────────────┬──────────────┐
  │ Protocole     │ MLP acc.     │ Err. moy.    │ CP α=10 %    │ Taille moy.  │
  ├───────────────┼──────────────┼──────────────┼──────────────┼──────────────┤
  │ Room-aware    │ 30.56 %      │ 0.944 m      │ 90.0 % cov.  │ 20.2 cellules │
  │ E102          │ 29.51 %      │ 0.994 m      │ 89.6 % cov.  │ 17.4 cellules │
  └───────────────┴──────────────┴──────────────┴──────────────┴──────────────┘

  Détail conformal :
  - Room-aware :
    * α=5 %  : couverture 95.1 %, taille moyenne 28.3 cellules
    * α=10 % : couverture 90.0 %, taille moyenne 20.2 cellules

  - La calibration conformal atteint bien les couvertures attendues :

    si la taille du jeu conformal est élevée ou si max-softmax est faible, on
    peut augmenter le lissage Kalman, demander une nouvelle mesure ou afficher
    un état "incertain" au lieu d'une position trop affirmative.

18.17 TCN-based Prototypical Network — Few-Shot Learning (Ma & Shi, Sensors 2023)
────────────────────────────────────────────────────────────────────────────────

  Référence :
  Ma Z., Shi K., "Few-Shot Learning for WiFi Fingerprinting Indoor Positioning",
  Sensors 2023, 23, 8458. https://doi.org/10.3390/s23208458

  Contexte et motivation :
  Ce papier propose d'utiliser le few-shot learning pour construire un modèle de
  localisation robuste avec très peu d'échantillons étiquetés. L'idée est d'éviter
  la collecte massive de données tout en gérant la fluctuation des signaux RSSI.
  Trois contributions principales :
    (1) Encodeur TCN (Temporal Convolutional Network) pour projeter le vecteur RSSI
        en espace d'embedding, capturant les relations spatiales entre antennes.
    (2) Prototype pondéré par confiance : chaque échantillon génère un vecteur
        embedding et un scalaire de covariance σ ; le prototype de classe est une
        moyenne pondérée par 1/σ (les mesures de bonne qualité pèsent plus).
    (3) Distance de Mahalanobis pour la classification (vs distance euclidienne
        classique des ProtoNet).
    (4) Filtre de Kalman en phase online pour débruiter les RSSI mesurés.

  Architecture TCN (Table 1 du papier) :
  ┌───────────┬──────────┬──────────┬──────────┬──────────┐
  │ Bloc       │ Kernel   │ Canaux   │ Dropout  │ Dilation │
  ├───────────┼──────────┼──────────┼──────────┼──────────┤
  │ Bloc 1     │ 5        │ 32       │ 0.20     │ 1        │
  │ Bloc 2     │ 5        │ 32       │ 0.20     │ 2        │
  │ Bloc 3     │ 5        │ 64       │ 0.50     │ 4        │
  │ Bloc 4     │ 5        │ 64       │ 0.50     │ 8        │
  │ Bloc 5     │ 5        │ 65       │ 0.50     │ 16       │
  └───────────┴──────────┴──────────┴──────────┴──────────┘
  65 = 64-d embedding + 1-d scalaire covariance brut S_raw.
  Covariance effective : σ = 1 + softplus(S_raw) — garantit σ > 1.

  Formules clés :
    Prototype :  p_c = Σ_i (s_i ⊙ x_i) / Σ_i s_i   avec s_i = 1/σ_i  [Éq. 4]
    Précision :  s_c = Σ_i s_i                                          [Éq. 5]
    Distance :   d_c(q) = √(s_c) × ‖q − p_c‖  (Mahalanobis scalaire)  [Éq. 6]
    Loss :       L = 1/N_Q Σ_i [−log( exp(−d_y(i)) / Σ_k exp(−d_k(i)) )]  [Éq. 8]

  Implémentation :
  Scripts : `scripts/tcn_protonet_wifi.py`   (v1 — TCN fidèle au papier)
            `scripts/tcn_protonet_wifi_v2.py` (v2 — 3 variantes encodeur)
  Protocole : LORO × K ∈ {1, 3, 5, 10, 25} échantillons support/cellule,
              5 trials par K, 2000–3000 épisodes méta-apprentissage/fold.
  Optimizer : Adam (lr=1e-3, weight_decay=1e-4) + CosineAnnealingLR.

  ── Analyse et diagnostic ──

  (a) Convergence du méta-apprentissage insuffisante :
      La loss finale oscille autour de 2.52–2.59 pour des tâches 15-way, alors
      que la loss aléatoire est ln(15) ≈ 2.71. Le modèle n'améliore que de ~7 %
      par rapport au hasard — signe d'un sous-apprentissage.

  (b) Inadéquation architecturale (cause principale) :
      Le TCN du papier est conçu pour des vecteurs RSSI de 8+ points d'accès
      ordonnés spatialement. Nos 5 features (Signal, Noise, A1, A2, A3) forment
      un vecteur de caractéristiques, PAS une séquence temporelle. Traiter ces
      5 valeurs comme une séquence de longueur 5 pour des convolutions causales
      dilatées (dilation jusqu'à 16) n'apporte aucun bénéfice :
        · dilation=1 → champ récepteur = 5 = seq_len (couvre déjà tout)
        · dilation=2,4,8,16 → redondant, la plupart du padding est zéro
      Conséquence : les blocs 2–5 n'apportent pas de couverture supplémentaire
      du signal d'entrée.

  (c) Accuracy décroissante avec K :
      Contrairement au papier (où plus K augmente, mieux le modèle fait), notre
      accuracy DÉCROÎT de K=1 (17.3 %) à K=25 (9.9 %). Cela vient du fait que
      le méta-apprentissage a été fait avec n_support=5 : pour K>5, le modèle
      n'a jamais vu de tâches avec autant de support, et la distance de
      Mahalanobis avec s_c ∝ K amplifie les distances de façon non calibrée.

  (d) Filtre de Kalman simulé trop bruité :
      La simulation online (bruit additif σ=3.0 dBm sur fenêtre T=10) ajoute
      plus de bruit que le KF ne peut en corriger, dégradant les résultats
      (acc KF ≈ 2 % vs 17 % sans KF).

  ── Comparaison avec la baseline ProtoNet-MLP (section 18.7) ──

  À K=1 : TCN-PN ≈ ProtoNet-MLP (1.056 m vs 1.059 m, +0.003 m gain TCN).
  À K>1 : ProtoNet-MLP est meilleur, car il maintient une accuracy ~14 %
           là où le TCN-PN chute à ~10 %.

  Conclusion v1 : le TCN-PN du papier ne surpasse pas le ProtoNet-MLP existant
  sur nos données 5D. La contribution de confiance (Éq. 4-6) ne peut pas
  compenser le sous-apprentissage dû à l'inadéquation architecturale.

  ── Analyse v2 ──

  (a) La pondération par confiance (MLP_conf) est validée :
      À K=1, MLP_conf ≈ MLP_plain (≈ 16.8 %). L'avantage devient majeur à K>1 :
      à K=25, MLP_conf gagne +4.6 pp et −53 mm vs MLP_plain. La raison : avec
      davantage d'échantillons support, la tête covariance identifie les mesures
      de mauvaise qualité et les sous-pondère dans le prototype, ce qui améliore
      le centroïde de classe. Cet effet est exactement celui prédit par le papier.

  (b) Le MLP_plain v2 corrige le bug de régression à K croissant du v1 :
      Avec n_support=10 (vs 5 en v1) et 3 000 épisodes (vs 2 000), la courbe
      K→accuracy est croissante comme attendu : 16.8 % → 24.3 % de K=1 à K=25.
      Le v1 montrait une décroissance due au mismatch train(n_support=5)/test(K≤25).

  (c) Le TCN_conf confirme l'inadéquation architecturale :
      Même avec la correction (last position vs avg-pool), la loss stagne à
      2.43–2.73 (barely better than random ≈ 3.00 for 20-way). L'accuracy
      plafonne à ~17 % indépendamment de K. Le TCN causal dilué apporte zéro
      valeur ajoutée sur des vecteurs de 5 features non ordonnés temporellement.

  (d) Filtre de Kalman simulé toujours pénalisant :
      Avec noise_std=2.0 dBm (réduit de 3.0→2.0 en v2), le KF reste néfaste
      (acc KF ≈ 3–6 % vs 17–29 % sans KF). La simulation ne peut pas reproduire
      la vraie dérive temporelle du signal : les données offline sont statiques,
      et le bruit artificiel ajouté ne correspond pas au processus réel.

  ── Intérêt pour l'article ──

  Le papier de Ma & Shi (Sensors 2023) apporte deux contributions exploitables :

    1. VALIDÉE — Pondération par confiance (Éq. 4-6) :
       MLP_conf surpasse MLP_plain de +4.6 pp / −53 mm à K=25 en LORO.
       Mécanisme robuste, simple à implémenter, compatible avec n'importe quel
       encodeur. C'est la contribution à retenir pour nos données 5D.

    2. NON TRANSFÉRABLE — Architecture TCN :
       Conçue pour des séquences RSSI multi-AP (≥8) ordonnées spatialement.
       Inadaptée à notre vecteur (Signal, Noise, A1, A2, A3). À éviter.

    3. NON EXPLOITABLE en l'état — Filtre de Kalman simulé :
       La simulation de fenêtre temporelle offline ne reflète pas les fluctuations
       RSSI réelles. Pertinent uniquement si des données online temps-réel sont
       disponibles (protocole de mesure séquentiel en phase de déploiement).

  Hiérarchie finale des méthodes few-shot LORO (K=25) :
    1. TargetKNN (§18.7)    : ~85 % acc  → gold standard few-shot supervisé
    2. MLP_conf (ce papier)  : 28.9 % acc, 0.926 m  ← meilleur ProtoNet
    3. MLP_plain v2          : 24.3 % acc, 0.979 m
    4. ProtoNet-MLP v1       : 13.7 % acc, 1.127 m
    5. TCN-PN v1             :  9.9 % acc, 1.146 m
    6. KNN-LORO (sans adapt.) :  2.3 % acc, 1.509 m  → plancher sans adaptation

  Conclusion : MLP_conf est recommandé comme méthode ProtoNet pour l'article.
  Il combine le mécanisme de confiance du papier avec un encodeur adapté aux
  données 5D, et offre une amélioration significative sur le ProtoNet vanilla
  quand K ≥ 3. La phrase clé pour l'article : "avec 25 mesures par cellule
  dans la salle cible, le ProtoNet à pondération confiance atteint 28.9 %,
  soit +11.9 pp vs ProtoNet standard, mais reste 56 pp sous le TargetKNN qui
  n'exige pas de méta-apprentissage inter-salles."

     KNN/RF pour la position, embeddings légers si besoin d'optimisation,
     conformal/selective pour la fiabilité, et TargetKNN/few-shot si une
     nouvelle salle dispose de mesures labellisées minimales.
