0. RÉSUMÉ EXÉCUTIF
════════════════════════════════════════════════════════════════════════════════

  Ce rapport documente l'ensemble des benchmarks réalisés pour un système de
  localisation indoor par fingerprinting RSSI Wi-Fi à granularité cellule
  (25 cm × 30 cm), sans capteur supplémentaire, dans 4 salles distinctes.

  CHIFFRES CLÉS POUR L'ARTICLE
  ──────────────────────────────

  Données :
  - 4 salles (B121, D005, E101, E102) | 29 730 échantillons total
  - 80/34/72/75 cellules | 5 features RSSI (Signal, Noise, A1, A2, A3)

  Meilleure précision (protocole room-aware, multi-salles, n=29 730) :
  - Top-1 (cellule exacte)    : 91.8 %  (KNN raw)      |  erreur moy. 0.097 m
  - Top-3 spatial             : 96.9 %  (NN+L-KNN)     |  top-3 min error 0.025 m
  - p90 erreur                : 0.00 m  → 90 %+ des prédictions sont à 0 cm
  - Précision distance (embedding NN+L-KNN) : 94.7 %

  Précision intra-salle (sans one-hot salle) :
  - E102 (75 cellules, 6 campagnes) : KNN = NN+L-KNN = 92.7 %
  - E101 (72 cellules, 6 campagnes) : KNN 91.0 %

  Généralisation — limites fondamentales :
  - LORO (salle inconnue)         : ~1–2 % tous modèles  → non transférable
  - LOCO E102 (campagne inconnue) : HistGB 2.42 %         → non transférable
  - LOCO B121 (campagne inconnue) : HistGB 2.30 %         → non transférable
  - B121/deplacement (inférence OOD/mouvement) : 100 % rejet OOD ; 0 % exact sur 6 points statiques
  - Conclusion : chaque déploiement requiert une recalibration complète.

  Robustesse :
  - Bruit synthétique σ=1 dBm : perte < 1 pp (KNN, LOCO E102)
  - Bruit σ=3 dBm             : dégradation significative (voir section 16septies)
  - Learning curve : 60 % des données suffisent pour atteindre ~90 % du plateau

  Protocole TriScope (modèle multi-tâche, D005+E101, 50 tirages) :
  - Cellule : 97.6 % ± 3.4 % | Distance routeur : 57.1 % | Salle : 72.1 %

  Calibration / Incertitude :
  - ECE KNN (brut) : 0.023 → très bien calibré naturellement
  - HistGB + temperature scaling : ECE 0.012, NLL 0.285 (meilleure calibration)

  Modèle recommandé en production :
  - Décisions "hard" (argmax)     → KNN (k=7, poids distance)
  - Décisions probabilistes       → HistGB + temperature scaling
  - Contrainte mémoire/latence    → voir section profil embarqué (16sexies/16quinquiesdecies)

  Référence état de l'art (Wi-Fi fingerprinting) :
  - Travaux publiés (KNN RSSI indoor) : 70–90 % @ grille 0.5–1 m typique
  - Ce système : 92–94 % @ cellule 25×30 cm → granularité ×20 plus fine

════════════════════════════════════════════════════════════════════════════════
1. CONTEXTE ET OBJECTIF
════════════════════════════════════════════════════════════════════════════════

L'objectif du projet est de réaliser une localisation indoor fine-grained (à
l'échelle de la cellule de 25 cm × 30 cm) en utilisant uniquement les signaux
Wi-Fi (RSSI) mesurés depuis un point d'accès fixe (routeur). Aucun capteur
supplémentaire n'est utilisé.

L'approche retenue est le fingerprinting RSSI : une phase hors-ligne ("offline")
collecte des mesures RSSI pour chaque position connue de la grille, constituant
une "carte radio". En phase en-ligne ("online"), le système classifie une
nouvelle mesure RSSI pour prédire la cellule occupée.

Matériel utilisé :
  - Routeur Wi-Fi standard avec 3 antennes distinctes (A1, A2, A3)
  - PC de mesure relié au routeur par câble RJ45 (distance routeur-PC : 125 cm)
  - Distance routeur–tableau blanc : 4 mètres, centré sur le tableau

Grille de mesure :
  - Cellules au format X_Y (ligne_colonne), taille 25 cm (largeur) × 30 cm (hauteur)
  - Lignes 0 à 4 (de devant vers le fond de la salle)
  - Colonnes de -4 à +10 (gauche à droite)
  - 75 cellules en E102, 35 cellules en D005, 34-72 cellules en E101


════════════════════════════════════════════════════════════════════════════════
2. FEATURES (VARIABLES D'ENTRÉE)
════════════════════════════════════════════════════════════════════════════════

Chaque échantillon est composé de 5 features brutes issues du scan Wi-Fi :

  1. Signal   : RSSI global du point d'accès (dBm)
  2. Noise    : niveau de bruit ambiant (dBm)
  3. signal_A1 : RSSI de l'antenne 1 du routeur (dBm)
  4. signal_A2 : RSSI de l'antenne 2 du routeur (dBm)
  5. signal_A3 : RSSI de l'antenne 3 du routeur (dBm)

Ces 5 features constituent le vecteur d'entrée des modèles de classification.
Dans les configurations "room-aware", un vecteur one-hot encodant la salle est
concaténé aux features RSSI. Dans le modèle NN+L-KNN, une couche d'embedding
apprend une représentation latente avant la classification KNN.

Nombre d'échantillons par cellule : 25 mesures par campagne.

4. MODÈLES DE MACHINE LEARNING ÉVALUÉS
════════════════════════════════════════════════════════════════════════════════

Tous les modèles résolvent un problème de classification multi-classes (75 classes
= 75 cellules) à partir du vecteur RSSI de 5 dimensions.

────────────────────────────────────────────────────────────────────────────────
4.1 K-Nearest Neighbors (KNN)
────────────────────────────────────────────────────────────────────────────────
  Principe : classe prédite = majorité parmi les k voisins les plus proches
  (distance euclidienne dans l'espace RSSI). Paramètre k optimisé par CV.
  Variantes testées :
    - KNN raw : KNN direct sur les 5 features
    - KNN distance+scaled : KNN pondéré par la distance après standardisation
      des features (baseline simple et robuste)
    - Bagging KNN : ensemble de KNN sur sous-échantillons bootstrap
    - KNN Mahalanobis : distance de Mahalanobis (skippé, données insuffisantes)
    - LDA+KNN : projection LDA puis KNN dans l'espace réduit

────────────────────────────────────────────────────────────────────────────────
4.2 NN + L-KNN (Neural Network + Latent KNN)
────────────────────────────────────────────────────────────────────────────────
  Architecture hybride en deux étapes :
    1. Un réseau de neurones (MLP) apprend un embedding latent des features RSSI
    2. Un KNN est appliqué dans cet espace latent pour la classification finale
  Avantage : capture les non-linéarités du signal RSSI tout en conservant la
  robustesse du KNN. Dans la configuration "cell+distance", le NN prédit
  simultanément la cellule ET la distance routeur.

────────────────────────────────────────────────────────────────────────────────
4.3 Random Forest (RF)
────────────────────────────────────────────────────────────────────────────────
  Ensemble de N arbres de décision entraînés sur des sous-échantillons bootstrap
  et des sous-ensembles de features. Robuste au sur-apprentissage.

────────────────────────────────────────────────────────────────────────────────
4.4 Extra Trees (ET)
────────────────────────────────────────────────────────────────────────────────
  Variante de Random Forest avec seuils de décision aléatoires (Extremely
  Randomized Trees). Généralement plus rapide que RF, précision comparable.

────────────────────────────────────────────────────────────────────────────────
4.5 Gradient Boosting (GBDT)
────────────────────────────────────────────────────────────────────────────────
  Ensemble séquentiel d'arbres corrigeant les erreurs du modèle précédent.
  Variantes :
    - GBDT classique (sklearn GradientBoostingClassifier)
    - HistGradientBoosting (HistGB) : version optimisée par histogrammes,
      inspirée de LightGBM, très rapide sur grands datasets

────────────────────────────────────────────────────────────────────────────────
4.6 Support Vector Machine (SVM)
────────────────────────────────────────────────────────────────────────────────
  Classifieur maximisant la marge entre classes. Noyaux testés :
    - SVM RBF (noyau gaussien)
    - SVM linéaire
    - SVM polynomial (degré 3)

────────────────────────────────────────────────────────────────────────────────
4.7 Deep MLP Softmax
────────────────────────────────────────────────────────────────────────────────
  Réseau de neurones profond (sklearn MLPClassifier) avec activation softmax
  en sortie pour la classification multi-classes.

────────────────────────────────────────────────────────────────────────────────
4.8 Modèles probabilistes
────────────────────────────────────────────────────────────────────────────────
  - GaussianNB : Naive Bayes gaussien (hypothèse d'indépendance des features)
  - QDA : Analyse Discriminante Quadratique
  - LDA : Analyse Discriminante Linéaire (utilisée comme prétraitement pour KNN)
  - Logistic Regression calibrée (Platt/sigmoid via CV) :
    baseline discriminative multi-classes avec probabilités calibrées.

────────────────────────────────────────────────────────────────────────────────
4.9 Modèle Multihead (RSSI seul, sans label de salle)
────────────────────────────────────────────────────────────────────────────────
  Architecture multi-tâche prédisant simultanément :
    - La cellule (classification à 75 classes)
    - La distance routeur (classification à 4 classes : 2/3/4/5 m)
    - La salle (classification à 3 classes : D005/E101/E102)
  Depuis le vecteur RSSI brut uniquement, sans one-hot salle.

────────────────────────────────────────────────────────────────────────────────
4.10 Modèles non disponibles (dépendances manquantes)
────────────────────────────────────────────────────────────────────────────────
  - CatBoost, LightGBM, XGBoost : skippés (non installés dans l'environnement)
  - Stacking (RF+SVM+MLP) : skippé dans les runs finaux (temps de calcul excessif,
    ~66 min par fold sur le dataset multi-salles)

────────────────────────────────────────────────────────────────────────────────
4.11 Piste hybride physique + ML (orientation Sionna RT)
────────────────────────────────────────────────────────────────────────────────
  Une piste complémentaire a été ajoutée pour relier modèle physique et
  adaptation data-driven, dans l'esprit Sionna RT (ray tracing différentiable).

  Implémentation actuelle :
    1) Modèle physique :
       - backend analytique (path-loss enrichi),
       - backend Sionna RT (ray tracing sur scènes intégrées, multi-antennes Tx).
    2) Calibration ML : régression Ridge multi-sortie pour corriger ces
       prototypes physiques à partir des mesures réelles.
    3) Inférence : appariement au prototype le plus proche (cellule + distance).

  Script dédié :
    - scripts/sionna_rt_hybrid.py
    - Sortie : reports/benchmarks/sionna_hybrid_metrics.json

  Commande :
    PYTHONPATH=src python scripts/sionna_rt_hybrid.py --protocol random --backend sionna

  Option Sionna RT (si disponible) :
    PYTHONPATH=src python scripts/sionna_rt_hybrid.py --protocol random --backend auto --use-sionna

  Remarque : si Sionna RT n'est pas installé, le script exécute tout de même le

  physique+ML avant intégration d'une scène Sionna complète (matériaux, géométrie,
  orientation d'antennes, et fine-tuning différentiable sur RSSI mesurés).

  Mise à jour expérimentale (exécutée le 12 mars 2026, backend Sionna RT actif) :

      * scénario custom-scene prêt dans le script, mais run complet non relancé
        dans cette itération (temps de calcul élevé).

  Fichiers produits :
    - reports/benchmarks/sionna_hybrid_metrics_random_sionna_customscene.json
    - (optionnel à relancer) reports/benchmarks/sionna_hybrid_metrics_loco_e102_sionna.json

  Interprétation rapide :
    - Le backend Sionna RT est désormais opérationnel (import OK), avec calcul

    - Une scène E102 custom a été reconstruite à partir des images du dossier

    - La grille de mesure est désormais projetée sur le plan du tableau

5. PROTOCOLES D'ÉVALUATION
════════════════════════════════════════════════════════════════════════════════

────────────────────────────────────────────────────────────────────────────────
5.1 Room-aware (intra-salle / multi-salle calibré)
────────────────────────────────────────────────────────────────────────────────
  Principe : toutes les données des salles disponibles sont mélangées, puis
  splitées 80% train / 20% test de façon stratifiée. Le modèle reçoit en entrée
  les features RSSI + un vecteur one-hot identifiant la salle.
  Interprétation : performance attendue en conditions réelles APRÈS calibration
  complète de toutes les salles. Le modèle connaît dans quelle salle il opère.
  Ce protocole ne teste PAS la généralisation à une nouvelle salle.

────────────────────────────────────────────────────────────────────────────────
5.2 LORO — Leave-One-Room-Out (généralisation inter-salles)
────────────────────────────────────────────────────────────────────────────────
  Principe : pour chaque salle R, entraîner sur toutes les autres salles,
  tester sur R. Avec l'ajout de B121, le protocole couvre désormais 4 folds
  (B121, D005, E101, E102 tour à tour).
  Interprétation : capacité du modèle à localiser dans une NOUVELLE salle non
  calibrée. C'est le vrai test de généralisation du fingerprinting RSSI.
  Limite fondamentale : les RSSI d'une salle inconnue sont incompatibles avec
  la carte radio d'une autre salle (propagation, géométrie, obstacles différents).

────────────────────────────────────────────────────────────────────────────────
5.3 LOCO — Leave-One-Campaign-Out (robustesse intra-salle)
────────────────────────────────────────────────────────────────────────────────
  Principe : dans une salle donnée, retirer une campagne entière, entraîner sur
  les autres et tester sur la campagne retirée.
  Interprétation : robustesse du modèle à un changement de condition opératoire
  au sein d'une même salle calibrée partiellement.
  Cas couverts dans ce rapport :
  - E102 : 6 campagnes (orientation antenne + élévation)
  - B121 : 3 campagnes (routeur centre/gauche/droite à 3 m)

────────────────────────────────────────────────════════════════════════════════
5.4 Classification de distance routeur
────────────────────────────────────────────────────────────────────────────────
  Tâche séparée : prédire la distance routeur–tableau (2/3/4/5 m) depuis le
  vecteur RSSI brut. Testé sur le dataset multi-salles (B121+D005+E101+E102).

────────────────────────────────────────────────────────────────────────────────
5.5 Classification de salle (room identification)
────────────────────────────────────────────────────────────────────────────────
  Tâche séparée : prédire la salle (B121/D005/E101/E102) depuis l'embedding appris
  par NN+L-KNN (sans label de salle). LogisticRegression sur l'embedding.

────────────────────────────────────────────────────────────────────────────────
5.6 Analyse spatiale (E102, RF 5-fold CV)
────────────────────────────────────────────────────────────────────────────────
  Évaluation cellule par cellule via cross-validation 5 folds sur E102 (RF).
  Permet d'identifier les zones géographiques difficiles à localiser.

────────────────────────────────────────────────────────────────────────────────
5.7 Protocole anti-leakage strict (à implémenter/valider)
────────────────────────────────────────────────────────────────────────────────
  Objectif : garantir qu'aucun quasi-duplicat (même cellule/campagne, RSSI très
  proche) n'est partagé entre train et test.

  Règles proposées :
  1) Split par groupes atomiques :
     - groupe = (room, campaign, grid_cell)
     - un groupe appartient intégralement au train OU au test, jamais aux deux.
  2) Vérification post-split obligatoire :
     - intersection exacte des groupes train/test = vide ;
     - intersection des signatures RSSI quantifiées (ex. pas 0.5 dB) sur un même
       (room, campaign, grid_cell) = vide.
  3) Agrégations calculées train-only :
     - toute feature agrégée (moyenne/variance/prototype par cellule/campagne,
       normalisation globale, etc.) est fit uniquement sur train puis appliquée
       au test ;
     - interdiction de recalculer une statistique en mélangeant train+test.
  4) Audit de fuite traçable :
     - journaliser `n_groups_train`, `n_groups_test`, `n_overlap_groups`,
       `n_overlap_signatures` ;
     - si overlap > 0, run invalide (échec explicite du benchmark).

  Bénéfice : métriques plus conservatrices mais robustes, et comparaison
  inter-modèles plus fiable en présence de séries RSSI fortement redondantes.

────────────────────────────────────────────────────────────────────────────────
5.8 Top-k spatial (ajout opérationnel)
────────────────────────────────────────────────────────────────────────────────
  En complément des métriques classiques (accuracy top-1, erreur moyenne),
  on suit désormais :
  - top-1 accuracy (`top1_acc`) ;
  - top-3 accuracy (`top3_acc`) ;
  - distance au meilleur candidat parmi le top-3 (`top3_best_error_m`) ;
  - gain spatial top-3 vs top-1 (`top3_best_gain_m`), en mètres.

  Intérêt opérationnel :
  même quand la cellule exacte top-1 est ratée, le top-3 peut rester
  géographiquement proche de la vérité terrain, ce qui reflète mieux
  la "rattrapabilité" de la décision en exploitation.

11quater. B121/DEPLACEMENT — CHALLENGE D'INFÉRENCE HORS ENTRAÎNEMENT
════════════════════════════════════════════════════════════════════════════════

Objectif :
  Évaluer directement le modèle d'inférence exporté sur un dossier explicitement
  exclu de l'entraînement (`data/B121/deplacement`) afin de tester sa capacité à
  "s'en sortir" sur des positions hors grille, des trajectoires en mouvement et
  des placements sémantiques (fond/centre/gauche/droite de la pièce).

Contenu évalué :
  - 6 points statiques : `0_15`, `2_-3`, `2_2`, `3_-8`, `3_9`, `4_12`
  - 10 traces d'inférence seules : `rond_411`, `diag_44`, `diag_4-4`,
    `hauteur_47`, `b_c`, `b_d`, `b_g`, `c_c`, `c_d`, `c_g`
  - 25 mesures RSSI par fichier, soit 16 traces × 25 = 400 inférences

Protocole :
  - Aucun ré-entraînement ; utilisation directe de `reports/localizer.joblib`
  - Inférence batch via `VertiLocInferenceModel`
  - Lecture des scores OOD déjà embarqués (énergie softmax + distance latente)

Résultats principaux :
  - Taux de rejet OOD : 100 % sur les 16 traces (400/400 échantillons rejetés)
  - Points statiques : 0 % de prédiction exacte sur les 6 cas
  - Erreur dominante sur les 6 points statiques : moyenne 1.884 m,
    médiane 1.583 m, max 3.856 m
  - Même le point interne `2_2` n'est pas récupéré exactement :
    cellule dominante prédite = `1_2` (erreur 0.30 m)
  - Les points hors tableau (`0_15`, `3_-8`, `4_12`) sont rabattus vers des
    cellules internes, jamais correctement identifiés comme position géométrique
    extrapolée
  - Les trajectoires (`rond`, `diag`, `hauteur`) et placements sémantiques
    (`b_*`, `c_*`) restent toutes rejetées comme hors distribution

Interprétation :
  Le modèle d'inférence se comporte ici comme un classifieur fermé sur sa carte
  radio d'entraînement : en dehors de la distribution calibrée, il projette les
  vecteurs RSSI vers des cellules connues mais le module OOD les rejette
  systématiquement. Autrement dit, sans nouvelle calibration, le système ne sait
  ni extrapoler spatialement hors grille, ni suivre proprement un mouvement
  continu à partir des seuls fingerprints statiques de B121.

Artefacts produits :
  - `scripts/evaluate_b121_deplacement.py`
  - `reports/benchmarks/b121_deplacement_eval.csv`
  - `reports/benchmarks/b121_deplacement_eval.json`
  - `reports/benchmarks/b121_deplacement_eval_predictions.csv`

Métriques de shift implémentées :
  - MMD RBF (Maximum Mean Discrepancy, estimateur non biaisé),
  - Wasserstein-1 (par feature + agrégats mean/max),
  - PSI simple (Population Stability Index, par feature + agrégats mean/max).

Implémentation et sorties :
  - Script : `scripts/shift_quantification.py`
  - Entrée perf : `reports/benchmarks/loco_e102.csv`
  - Sorties :
      `reports/benchmarks/shift_loco_e102.csv`
      `reports/benchmarks/shift_loco_e102_perf_corr.csv`
      `reports/benchmarks/shift_loco_e102_perf_corr.json`
  - Figure article :
      `reports/benchmarks/shift_loco_e102_overview.png`
  - Tableau compact (top corrélations) :
      `reports/benchmarks/shift_loco_e102_top_correlations.csv`

Faits saillants (sur 6 folds LOCO) :
  - Le split le plus décalé est exp6_1m50 (routeur surélevé), avec les plus
    fortes valeurs de divergence :
      MMD = 0.1827, Wasserstein_mean = 3.335, PSI_mean = 0.958, PSI_max = 2.003.
  - Les splits exp2/exp4/exp5 présentent des divergences plus faibles
    (MMD ≈ 0.019–0.022).
  - L'ordre de sévérité du shift est cohérent avec l'intuition physique :
    élévation forte (exp6) > changements d'orientation simples.

Corrélation shift -> dégradation de performance :
  - Cible 1 : `acc_drop` = max(acc) - acc (par modèle, par fold),
  - Cible 2 : `err_increase` = err - min(err) (par modèle, par fold).

Exemples de corrélations observées (Pearson) :
  - ExtraTrees, `err_increase` vs `psi_max` : r = +0.820
  - ExtraTrees, `err_increase` vs `mmd_rbf` : r = +0.788
  - HistGB,    `err_increase` vs `psi_max` : r = +0.705

Interprétation :
  - Sur plusieurs modèles (notamment ExtraTrees/HistGB), plus le shift
    train/test est élevé, plus l'erreur augmente.
  - Le signal est net pour l'erreur moyenne, plus instable pour l'accuracy
    cellule (normal sur un problème à 75 classes déjà proche du hasard en LOCO).
  - Taille d'échantillon faible (n=6 folds) : ces corrélations sont des indices
    mécanistiques utiles, pas une preuve statistique définitive.

16. SYNTHÈSE ET CONCLUSIONS
════════════════════════════════════════════════════════════════════════════════

────────────────────────────────────────────────────────────────────────────────
14.1 Performance intra-salle (conditions calibrées)
────────────────────────────────────────────────────────────────────────────────
  Les meilleures approches (Bagging KNN, LDA+KNN, NN+L-KNN, HistGB) atteignent
  93–94 % de précision cellule avec erreur <9 cm sur les multi-salles calibrées,
  et 92–93 % sur E102 seul. Le p90=0 confirme que la majorité des prédictions
  sont exactes au cm près.

────────────────────────────────────────────────────────────────────────────────
14.2 Limite fondamentale : non-transférabilité inter-salles
────────────────────────────────────────────────────────────────────────────────
  Le LORO démontre que TOUS les modèles échouent (~1–2 %) lorsque la salle
  cible n'a pas été calibrée. L'empreinte RSSI est propre à chaque environnement
  physique. Cette limitation est inhérente au fingerprinting et non aux modèles.

────────────────────────────────────────────────────────────────────────────────
14.3 Sensibilité aux conditions opératoires (LOCO)
────────────────────────────────────────────────────────────────────────────────
  Au sein d'E102, un changement d'orientation d'antenne ou d'élévation du
  routeur est aussi dévastateur qu'un changement de salle (~1–2 %). La campagne
  au sol (exp5) est particulièrement incompatible avec les autres conditions.
  → Pour un déploiement robuste, il est nécessaire de couvrir toutes les
    configurations opératoires dans les données de calibration.

────────────────────────────────────────────────────────────────────────────────
14.4 Hiérarchie des modèles
────────────────────────────────────────────────────────────────────────────────
  TIER 1 (≥93 %) : Bagging KNN, LDA+KNN, NN+L-KNN, HistGB, KNN raw
  TIER 2 (90–93 %): ExtraTrees, RandomForest
  TIER 3 (83–90 %): GradientBoosting
  HORS-CLASSE (<50 %): SVM (RBF/poly/lin), Deep MLP sklearn, GaussianNB, QDA

  Les modèles à noyau (SVM) et les modèles purement paramétriques (MLP sklearn,
  GaussianNB) sont inadaptés à ce problème de classification RSSI à 75 classes.

────────────────────────────────────────────────────────────────────────────────
14.5 Recommandations pour les prochaines campagnes
────────────────────────────────────────────────────────────────────────────────
  1. Inclure systématiquement TOUTES les orientations d'antenne prévues en
     déploiement dans le dataset de calibration (le LOCO le confirme).
  2. L'élévation (au sol vs. standard vs. surélevé) requiert des campagnes
     dédiées si le routeur peut être placé à différentes hauteurs.
  3. La cellule 1_-4 (coin gauche, ligne 1) est systématiquement difficile :
     envisager une densification de la grille dans les coins.
  4. Modèles recommandés : KNN-based (Bagging KNN ou LDA+KNN) pour leur
     simplicité et leur performance maximale. NN+L-KNN pour les scénarios
     où la distance doit aussi être estimée (91.9 % de précision distance).

────────────────────────────────────────────────────────────────────────────────
14.6 Piste complémentaire : GAN / CycleGAN / estimation générative de carte radio
────────────────────────────────────────────────────────────────────────────────
  Les approches génératives récentes (GAN conditionnels, CycleGAN, generative
  radio map estimation) constituent une piste crédible pour réduire le coût de
  recalibration lorsqu'une nouvelle salle dispose de peu de mesures.

  Intérêt principal :
  - Générer/compléter des fingerprints RSSI manquants à partir d'un sous-ensemble
    de points mesurés.
  - Effectuer un transfert de domaine (salle source -> salle cible) en apprenant
    une transformation de distribution RSSI.

  Limites pratiques (premier déploiement) :
  - Entraînement instable et sensible au tuning (mode collapse, variabilité run-to-run).
  - Risque d'halluciner des structures radio non physiques si la contrainte
    géométrique/proagation n'est pas suffisamment imposée.
  - Complexité d'implémentation/validation nettement supérieure à RadioUNet,
    XGBoost ou un GNN bien régularisé.

  Position recommandée pour ce projet :
  - Prioriser d'abord un socle robuste et reproductible (RadioUNet, XGBoost,
    GNN) comme baseline opérationnelle.
  - Évaluer ensuite la piste générative comme module d'augmentation/transfert,
    avec protocole strict :
      (i) scénario low-data cible (5-20 % de points mesurés),
      (ii) comparaison contre interpolation + baselines discriminatives,
      (iii) métriques cellule + erreur spatiale + stabilité inter-seeds,
      (iv) ablation avec/sans contraintes physiques.

  Conclusion opérationnelle : piste prometteuse pour le few-shot inter-domaines,
  mais plus risquée qu'une approche discriminative classique en premier essai.

14.7 Essais implémentés (code) — cGAN/CycleGAN sur E102 (low-data)
────────────────────────────────────────────────────────────────────────────────
  Implémentation réalisée : script `scripts/generative_radio_map.py` avec deux
  modes (`cgan`, `cyclegan`) et protocole d'évaluation standardisé :
  - baseline : KNN entraîné sur faible quantité de données cible
  - augmenté : KNN entraîné sur (données réelles low-data + données synthétiques)
  - upper bound : KNN entraîné sur train cible complet

  Paramétrage principal des essais :
  - cGAN : 40 epochs, 20 échantillons synthétiques/cellule
  - CycleGAN : 20 epochs, transfert exp2 -> cible, 20 synthétiques/cellule
  - split cible : 80/20, low-data = 10 % ou 20 % du train

  Résumé chiffré (delta = augmenté - baseline) :
  - cGAN (6 runs) :
    delta_acc moyen  = -0.09 points
    delta_err moyen  = +0.005 m
    Observation : effet global quasi neutre, parfois légère amélioration locale
    de l'erreur (ex: exp1 à 10 %, seed 7 : -0.005 m), mais non stable selon seed.
  - CycleGAN (3 runs) :
    delta_acc moyen  = -4.53 points
    delta_err moyen  = +0.066 m
    Observation : dégradation systématique de la précision cellule ; transfert
    de style RSSI inter-campagne insuffisamment fiable dans cette configuration.

  Stabilité statistique recommandée (idée à explorer en protocole standard) :
  - Répéter chaque protocole avec au moins 10 seeds indépendantes.
  - Reporter, pour chaque métrique (accuracy, erreur moyenne, delta), le triplet
    moyenne ± écart-type, puis l'IC95% :
      IC95% = moyenne ± t_(0.975, n-1) * (ecart_type / sqrt(n))
  - Interprétation : un gain n'est considéré robuste que si l'IC95% du delta
    n'inclut pas 0.
  - Sortie dédiée ajoutée dans le code : `generative_trials_stats.csv`
    (agrégation par protocole avec n_seeds, moyenne, std, bornes IC95%).
  - Exécution recommandée (10 seeds) :
    `python3 scripts/super_benchmarks.py --run-generative --generative-profile full --generative-seeds 1-10`

  Conclusion empirique à ce stade :
  - La piste générative est bien implémentée et testée, mais ne dépasse pas les
  baselines discriminatives sur les essais actuels.
  - cGAN peut être conservé comme module exploratoire (effet faible/neutre),
    tandis que CycleGAN est à considérer comme trop risqué sans contraintes
    physiques additionnelles ou tuning plus poussé.

16septdecies. BENCHMARK AUXILIAIRE "ÉLÉVATION" (E102)
════════════════════════════════════════════════════════════════════════════════

Objectif :
  Tester explicitement si les 5 RSSI contiennent une signature exploitable de
  la hauteur du routeur, en ajoutant une tâche auxiliaire supervisée
  `elevation_class` au modèle cellule.

Définition des classes d'élévation :
  - 0.00 m  : exp5, routeur posé au sol
  - 0.75 m  : exp1 à exp4, hauteur nominale
  - 1.50 m  : exp6, routeur posé sur deux tables (2 × 75 cm)

Protocole :
  - Dataset : E102 complet (11 250 échantillons, 75 cellules, 6 campagnes)
  - Entrées : uniquement les 5 features RSSI
  - Split : 5 tirages stratifiés 80/20 sur (cellule, elevation_class)
  - Baseline : MLP cellule seule
  - Multi-tâche : MLP à tronc partagé + tête cellule + tête élévation

Résultats globaux — tâche élévation :
  - Accuracy moyenne  : 70.11 %  (std 0.59 pp)
  - F1 macro moyen    : 0.458    (std 0.019)

Matrice de confusion agrégée (5 seeds, comptes bruts) :

                  Prédit 0.00m   Prédit 0.75m   Prédit 1.50m
  Vrai 0.00m            95            1 628           152
  Vrai 0.75m            59            7 058           383
  Vrai 1.50m            17            1 124           734

Lecture :
  - La classe nominale 0.75 m est reconnue correctement dans la majorité des cas.
  - En revanche, les hauteurs extrêmes sont souvent rabattues vers 0.75 m.
  - La tête auxiliaire détecte donc bien une structure liée à la hauteur, mais
    pas assez nettement pour distinguer proprement exp5 (sol) et exp6 (1.50 m)
    à partir des seuls 5 RSSI dans ce protocole.

Performance par campagne (au minimum exp5 / exp6) :
  - exp5_ground (0.00 m) : accuracy élévation = 5.1 %
  - exp6_1m50  (1.50 m)  : accuracy élévation = 39.1 %
  - exp1_back_right      : 98.6 %
  - exp2_front_right     : 87.4 %
  - exp3_front_left      : 92.3 %
  - exp4_back_left       : 98.2 %

Impact sur la tâche cellule :
  - Baseline cellule seule       : 16.14 % accuracy moyenne
  - Modèle multi-tâche cellule+élévation : 16.33 % accuracy moyenne
  - Gain absolu moyen            : +0.19 point
  - F1 macro cellule : 0.146 → 0.147

Conclusion scientifique :
  - L'ajout de la tête élévation n'apporte qu'un gain marginal sur la cellule.
  - Le signal RSSI porte une information de hauteur partielle, dominée ici par
    la classe nominale 0.75 m.
  - Dans l'état actuel de ce dataset, cette expérience ne suffit pas à elle
    seule pour démontrer que l'élévation est le facteur principal du shift ;
    elle suggère plutôt un effet couplé hauteur + orientation + contexte de
    campagne.
  - Pour renforcer cette thèse, il faudrait un protocole contrôlé où seule la
    hauteur varie, avec davantage de niveaux intermédiaires et/ou des features
    plus riches que 5 RSSI scalaires.

16octodecies. BENCHMARK AUXILIAIRE PYTORCH — NN+L-KNN
════════════════════════════════════════════════════════════════════════════════

Objectif :
  Vérifier la faisabilité d'une variante PyTorch du modèle NN+L-KNN, distincte
  du pipeline principal actuellement implémenté avec `sklearn MLPClassifier`.

Implémentation :
  - Script : `scripts/nn_lknn_torch.py`
  - Encoder : MLP PyTorch (5 entrées RSSI -> 64 -> 32 -> softmax)
  - Décodeur : KNN local sur l'embedding latent (k=5, poids distance)
  - Optimisation : Adam, lr=1e-3, weight_decay=1e-4
  - Prétraitement : `StandardScaler`
  - Environnement d'exécution : `.venv` locale, torch 2.10.0+cpu
  - Date d'exécution : 19 mars 2026

Protocole :
  - Split unique stratifié 75/25 (`test_size=0.25`)
  - Seed fixe : 42
  - CPU uniquement, threads bornés (`OMP_NUM_THREADS=1`)
  - Attention : ces runs sont auxiliaires et ne remplacent pas les tableaux
    principaux multi-seeds / room-aware du rapport.

Résultats auxiliaires obtenus :

  1) D005 (ddeuxmetres + dquatremetres, run par défaut du script)
     - train = 1 275 | test = 425
     - Accuracy cellule : 98.35 %
     - Erreur moyenne   : 0.0119 m

  2) E102 complet (exp1..exp6, distance 4 m)
     - train = 8 437 | test = 2 813
     - Accuracy cellule : 91.50 %
     - Erreur moyenne   : 0.1260 m

Lecture :
  - La variante PyTorch fonctionne correctement et retrouve un niveau de
    performance cohérent avec le pipeline `sklearn` existant.
  - Sur D005, le run rapide atteint 98.35 %, très proche des meilleurs résultats
    intra-salle déjà rapportés (~98 %).
  - Sur E102 complet, le run rapide atteint 91.5 %, légèrement sous les ~92.7 %
    du benchmark principal intra-salle, mais dans le même ordre de grandeur.
  - L'écart résiduel peut s'expliquer par le protocole plus simple ici
    (un seul split, 30 epochs, pas d'agrégation multi-seeds, pas de calibration
    ni de recherche d'hyperparamètres).

Conclusion pratique :
  - Oui, une version PyTorch de NN+L-KNN a été exécutée avec succès.
  - À ce stade, elle constitue une preuve de faisabilité technique crédible,
    mais pas encore un remplacement méthodologiquement strict du pipeline
    `sklearn` utilisé dans les sections comparatives principales.
